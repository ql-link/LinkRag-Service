# 测试驱动开发

## 概述

先写测试，让它失败，再写最小代码让它通过。

**核心原则：** 如果你没有亲眼看到测试先失败，就不知道它是否真的测到了该测的行为。

**违反规则字面要求，就是违反规则本意。**

## 所需上下文

开始前至少准备：

1. 当前需求或缺陷定义
2. 需要证明的行为
3. 当前测试框架和既有测试风格

按需补读：

4. 相关实现代码
5. 仓库中的相似测试
6. 现有失败场景或复现脚本

## 进入条件

如果你已经先写了生产代码，再想补测试，那顺序已经错了。这个 skill 默认你遵守“先测试后实现”。

## 铁律

```text
没有失败测试，就不能写生产代码
```

如果你已经先写了代码，那就删掉，重新来。

## 红-绿-重构

### 红：先写失败测试

写一个最小测试，只验证一个行为。

要求：

- 一次只测一个行为
- 名称清晰
- 尽量测真实行为，不要只测 mock

### 验证红：亲眼看到它失败

这是强制步骤，不能跳过。

确认：

- 测试确实失败，而不是报错
- 失败原因符合预期
- 是因为功能未实现，而不是因为测试本身写错

如果测试直接通过，说明你没有测到新行为。

### 绿：写最小实现

只写刚好让当前测试通过的最小代码。

不要顺手加功能，不要顺手重构，不要顺手优化。

### 验证绿：亲眼看到它通过

再次运行测试，确认：

- 当前测试通过
- 相关测试没有被打坏

### 重构：在全绿前提下清理代码

只有在测试全绿后，才允许：

- 去重
- 改善命名
- 提取辅助方法

重构后必须保持测试仍然通过。

## 好测试的标准

| 维度 | 好测试 | 差测试 |
|------|--------|--------|
| 最小化 | 一次只测一个行为 | 一个测试塞多个行为 |
| 清晰度 | 名称直接表达行为 | `test1` 之类无意义名称 |
| 意图表达 | 能看出预期 API 和预期行为 | 只看到实现细节 |

## 为什么顺序不能反

“我先写代码，最后补测试也一样”是错的，因为：

- 你没证明测试真的能抓到问题
- 很容易测到实现细节而不是行为
- 很容易漏掉边界条件
- 你永远没见过它真正失败的样子

“我手工测过了”也不等于有回归保护，因为：

- 手工测试不可重复
- 没有记录
- 后续代码变了不能自动复跑

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- 先写了失败测试
- 确认它因为正确原因失败
- 只加了最小生产代码让它通过
- 重构后测试仍然全绿

## 红旗信号 - 立即停止

- 先写实现代码，把它当“草稿”
- 一次写多个行为的测试
- 接受“测试报错”而不是“测试正确失败”
- 当前测试只要求一点，却扩写很多实现
- 因为“太明显了”而跳过红绿验证

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| “实现完再补测试” | 那你没有证明测试真的能抓问题。 |
| “这个太简单，不用 TDD” | 越简单越应该容易守住纪律。 |
| “先手工验证一下就行” | 手工验证不能替代自动化回归。 |
| “代码已经写了一大半了” | 沉没成本不能把错误顺序变正确。 |
| “我写个大而全的测试更省事” | 大测试会掩盖行为边界，调试更慢。 |

## 最终结论

`test-driven-development` 的职责是让行为在实现前就变成可执行约束。

**如果你没有先看到测试失败，你就还不能真正信任这个测试。**
