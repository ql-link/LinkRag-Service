# 技术设计

## 概述

这个阶段的目标不是“把 PRD 再复述一遍”，而是把已经审核通过的需求，转换成**能指导实现、能约束边界、能被审核放行**的 `technical_design.md`。

本 skill 主要回答：

- 准备改哪些模块
- 复用哪些现有能力
- API、数据、缓存、消息、对象存储如何设计
- 异常类、异常响应和错误码如何定义
- 风险、兼容性、测试策略是什么

它不直接负责：

- 写业务代码
- 产出改造结果
- 输出测试执行结论

这些分别属于后续 `implementation-execution` 和 `test-and-delivery`。

**核心原则：** 技术方案必须建立在真实代码和真实组件能力之上，而不是建立在猜测之上。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始前至少读取：

1. `AGENTS.md`
2. `project_info.md`
3. 当前模块当前期次目录 `feature_info.md`
4. 当前模块当前期次目录 `requirement.md`
5. `.agents/skills/technical-design/technical_design.template.md`
6. `.agents/skills/middleware-contracts/SKILL.md`
7. `docs/组件和数据库约定/middleware_contract.md`

按需补读：

8. 对应组件说明文档
9. 同业务域历史模块目录中的 `technical_design.md`
10. 同业务域历史模块目录中的 `implementation_report.md`
11. 相关真实代码

组件文档读取规则：

- 涉及 Redis 缓存时读 `redis_component.md`
- 涉及文件存储、公私有访问、object key 时读 `oss_component.md`
- 涉及异步消息、topic、consumer 时读 `kafka_component.md`

## 进入条件

只有满足以下条件时，才允许进入本 skill：

1. 已完成 `project-bootstrap`
2. 当前模块目录与当前期次目录存在
3. `requirement.md` 已经过人工审核通过
4. 当前复杂度等级为 `L2` 或 `L3`，或用户明确要求单独技术方案

如果需求文档还未审核通过，就不要开始写 `technical_design.md`。

## 输出

输出文件固定为：

`docs/模块开发文档/<模块名称>/<phase>/technical_design.md`

同时必须同步维护：

- 当前模块当前期次目录 `feature_info.md`

`technical_design.md` 必须遵循：

- `.agents/skills/technical-design/technical_design.template.md`

## 阶段一：校验设计是否就绪

在正式写方案前，先确认以下条件：

1. 需求边界已稳定，不再处于讨论状态
2. 相关模块入口已定位
3. 相关组件文档已定位
4. 涉及的公共契约已读取
5. 当前实现是否需要兼容、迁移、扩展 framework 已有初步判断

### 阶段一输出

如果仍存在关键空白，只允许输出：

1. 当前已确认的设计前提
2. 当前仍待确认的技术问题
3. 为什么这些问题会阻塞 `technical_design.md`

此时必须继续提问，不允许直接写完整方案。

## 阶段二：扫描真实代码

在写方案前，必须先扫描真实代码。

至少确认：

1. 当前模块入口在哪里
2. 相近功能怎么写
3. 是否已有可复用组件或服务
4. 是否已有同类 DTO / Entity / Mapper / Controller / 配置类

补充要求：

- 若方案中提到“复用某组件”，必须说明读过哪个组件文档
- 若方案中提到“复用某模块现有实现”，必须说明参考了哪些代码文件
- 若方案中提到“新增代码放在哪个包”，必须基于现有包结构判断，不能空想

## 阶段三：编写技术设计文档

当上下文足够明确后，按模板写 `technical_design.md`。

最低要求必须覆盖：

1. 文档修订记录
2. 技术目标与范围
3. 当前系统分析
4. 总体方案设计
5. API 设计
6. 数据与存储设计
7. 核心实现设计
8. 组件与集成设计
9. 权限、安全与审计
10. 异常类与错误码设计
11. 异常处理与降级策略
12. 测试方案
13. 发布方案

写作规则：

- 方案必须引用当前仓库真实代码，不允许空想
- 必须区分业务层改动与 framework 改动
- 需要说明复用哪些已有能力，以及为什么这样复用
- 需要明确模块边界、数据边界、组件边界
- 需要明确哪些内容留在本期、哪些内容后置
- 涉及接口时，必须明确异常类、错误码、HTTP 状态和统一响应边界
- 涉及前端对接接口时，响应结构必须按接口或响应 DTO 分组写清楚，至少覆盖成功响应、分页响应、空响应、关键展示字段和前端判断字段；不要只放一个 happy path 示例
- 前端响应示例默认只写展示和业务判断必需字段；不要暴露服务端内部定位字段、对象存储 bucket/object key、内部 URL、Token、密钥、异常堆栈或其他敏感信息。若确需前端访问文件，应单独设计带权限校验的下载/预览接口
- 涉及异常类和错误码时，必须先参考现有 `BusinessException`、错误码枚举、统一返回结构和已有异常处理代码，保持命名、放置位置、继承关系和编码风格一致
- 涉及 Redis、OSS、MQ 等中间件时，必须先读取对应中间件组件说明文档，再决定代码放置位置、接入方式和扩展边界
- 涉及多张数据表字段设计时，必须按表拆成独立小节或独立表格书写；不要把多张表的字段混在同一张字段表中，避免职责边界和实现归属混淆

## 阶段四：更新 feature_info

写完 `technical_design.md` 后，必须同步更新 `feature_info.md`，至少回填：

- 当前状态：`技术设计中` / `技术方案待审核`
- 当前实际产出的文档清单
- 当前推荐阅读顺序

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- `technical_design.md` 已写完
- `feature_info.md` 已同步更新
- 当前文档已进入“技术方案待审核”状态
- 已明确告知用户下一步是审核技术方案

## 红旗信号 - 立即停止

如果你发现自己开始做以下事情，说明已经跑偏，必须停下来：

- 凭类名猜测现有实现
- 凭经验假设组件能力
- 没读代码就规划模块边界
- 一边写技术方案，一边直接写实现结果
- 把需求文档原文整段搬进技术文档
- 用“按现有逻辑扩展”代替真实代码引用

这些都属于**脱离代码现实写方案**。

## 常见自我合理化

| Excuse | Reality |
|--------|---------|
| “这个模块我大概记得，不用再读代码” | 记忆不等于现状，必须读真实代码。 |
| “先把方案写个大概，后面实现时再修” | 技术方案一旦脱离代码，后面实现只会不断返工。 |
| “组件应该支持这个能力” | 应该不等于已经支持，先读组件文档和实现。 |
| “包结构先按常见 Spring 分层写” | 新增代码位置必须基于仓库现状判断。 |
| “这块先写成以后再补” | 模糊技术方案不能指导实现，也不能有效审核。 |

## 最终结论

`technical-design` 的职责是：**把已审核通过的需求，转换成建立在真实代码基础上的工程方案。**

**没有代码引用，就没有合格的技术方案。**
