# 系统化调试

## 概述

随机修补只会浪费时间，还会引入新问题。快速补丁往往只是遮住症状，没有触到根因。

**核心原则：** 在确认根因之前，不允许直接提修复方案。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始前至少准备：

1. 报错信息、栈追踪、失败测试或异常现象
2. 最近相关代码或配置改动
3. 当前已知的复现步骤

按需补充：

4. 相关日志
5. 相关测试
6. 相关配置和环境传递链路
7. 同仓库中的可工作的对照实现

## 铁律

```text
未完成根因调查前，不允许直接修复
```

如果你还没有完成第一阶段，你就没有资格提修复方案。

## 进入条件

如果你还不知道根因，你仍然处在“调试”阶段，而不是“修复”阶段。

## 四个阶段

### 阶段一：根因调查

在尝试任何修复前，先做以下事情：

1. 仔细阅读错误信息
   - 不要跳过 warning 和 error
   - 完整阅读栈追踪
   - 记下行号、文件路径、错误码

2. 稳定复现
   - 能否稳定触发
   - 触发步骤是什么
   - 是否每次都出现
   - 如果不能稳定复现，就继续收集证据，不要猜

3. 检查最近改动
   - 最近改了什么可能导致这个问题
   - 看 git diff、最近提交、依赖变更、配置变更、环境差异

4. 多组件系统先补诊断信息
   - 在每个组件边界记录输入和输出
   - 验证环境变量或配置是否真的传递到下一层
   - 先定位坏在哪一层，再深入那一层

5. 追踪数据流
   - 错值从哪里来的
   - 是谁把它传下来的
   - 持续向上追，直到找到真正源头

### 阶段二：模式分析

在修复前先找到模式：

1. 找可工作的对照实现
2. 读完整参考实现，不要只扫一眼
3. 列出可工作和不可工作的差异
4. 理解依赖前提、配置假设和调用链

### 阶段三：提出假设并验证

按科学方法来：

1. 形成单一假设
   - 明确写出“我认为根因是 X，因为 Y”

2. 做最小化验证
   - 一次只改一个变量
   - 用最小改动验证假设

3. 验证结果后再继续
   - 成立就进入修复阶段
   - 不成立就回到调查阶段，形成新假设

4. 如果你不懂
   - 直接承认“不理解这里”
   - 不要装懂，不要猜

### 阶段四：实施修复

只修根因，不修症状：

1. 先写最小失败用例
2. 再做单一修复
3. 验证问题真的消失且没有引入回归
4. 如果连续多次修复失败，回头质疑架构本身

## 红旗信号 - 立即停止

如果你开始这么想，说明已经跑偏：

- “先快速修一下，之后再查”
- “先试着改改看”
- “多改几个地方一起跑”
- “先不写测试，我手工看一下”
- “大概就是这里的问题”
- “虽然没完全看懂，但这个改法应该能行”
- “已经试了很多次，再多试一次”

这些都意味着：停止，回到根因调查。

## 你的合作方在提醒你跑偏的信号

- “这个真的发生了吗？”：说明你在没验证前做假设
- “这样能证明问题在哪一层吗？”：说明你缺证据
- “别猜”：说明你在提修复而不是做调查
- “认真想清楚”：说明应该回头质疑系统结构，而不是继续打补丁

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- 已用证据定位到可信根因
- 已缩小到具体故障边界或组件
- 已形成可以被最小改动验证的明确假设
- 已准备好在不靠猜测的前提下进入修复

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| “我大概知道是哪里” | 没追踪过，就还不知道。 |
| “先试一个小修复” | 快速猜测会换来更慢返工。 |
| “应该是下游的问题” | “应该”不是证据。 |
| “先上线修复，之后再查” | 把猜测上线不是调试。 |
| “这是偶发问题，不好查” | 越偶发越需要更多证据，而不是更少。 |

## 最终结论

`systematic-debugging` 的职责是强制先找根因，再做修复。

**如果你在证据之前提修复，那你不是在调试，而是在碰运气。**
