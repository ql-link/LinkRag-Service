# 中间件契约

## 概述

本 skill 的职责不是替代规则文档本身，而是告诉 AI 和开发者：

- 什么时候必须读取项目的中间件与跨模块约定
- 什么时候只是复用既有规则
- 什么时候已经形成新的公共契约
- 什么时候必须回写规则文档

它主要服务于：

- `technical-design`
- `implementation-execution`
- `review-and-quality`

**核心原则：** 规则文档负责记录规则，本 skill 负责驱动规则的使用时机和更新时机。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始前至少读取：

1. `AGENTS.md`
2. `project_info.md`
3. `docs/architecture/middleware_contract.md`

按需补读：

4. 当前模块当前期次目录 `feature_info.md`
5. 当前模块当前期次目录 `technical_design.md`
6. 当前模块当前期次目录 `implementation_report.md`
7. 对应组件说明文档
8. 当前涉及的真实代码

## 进入条件

在以下任一场景中，必须使用本 skill：

1. 进入技术设计阶段，且设计涉及数据库、缓存、消息、对象存储、统一接口或跨模块规则
2. 进入代码实现阶段，且实现可能新增或修改公共规则
3. 需要判断某项变更到底是“业务内部实现细节”还是“项目级公共契约”

如果当前任务完全局限于单个业务模块内部，且不会形成复用规则，可以不单独运行本 skill。

## 输出

本 skill 的输出不是新文档，而是明确结论：

1. 本次是否需要读取 `docs/architecture/middleware_contract.md`
2. 本次是“复用现有约定”还是“新增/修改公共契约”
3. 若存在新规则，应回写到 `docs/architecture/middleware_contract.md` 的哪一节

## 阶段一：识别触面

先判断当前需求或改动是否触碰以下任一公共面：

- MySQL 表结构、公共字段或命名规则
- Redis key 命名、TTL 或一致性策略
- MQ topic、group、消息结构或消费边界
- OSS bucket、object key、公私有访问规则
- 统一 API 响应、错误码、日志或追踪规则

如果没有触碰以上任一公共面，通常不需要进一步使用本 skill。

## 阶段二：区分复用和新增

如果已触碰公共面，继续判断：

1. 当前只是复用现有规则
2. 当前引入了新的公共规则
3. 当前修改了既有公共规则的含义或边界

判断标准：

- 只在单个业务里生效、不会被别处遵守的，通常不算公共契约
- 只要未来其他模块需要沿用、兼容或引用，就属于公共契约

## 阶段三：守门与回写规则文档

若确认形成了新的公共契约，必须：

1. 指出应更新 `docs/architecture/middleware_contract.md` 的具体章节
2. 在技术方案中说明新规则与业务边界
3. 在实现或改造报告中记录实际落地情况
4. 在评审阶段把“是否已同步回写公共契约”作为检查项

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- 已判断本次是否触碰公共契约
- 已区分“复用现有约定”还是“新增/修改约定”
- 若存在契约变化，已明确指出需要更新规则文档

## 红旗信号 - 立即停止

- 不读规则文档就新增 Redis key、MQ topic、OSS 路径或公共字段
- 把单次业务写法误升格成项目级规则
- 新规则已经进入代码，却不回写规则文档
- 把组件实现细节错误写进规则手册
- 发现公共规则已变化，却仍然把它当成“业务内部细节”处理

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| “只是顺手加一个 key，不算公共约定” | 只要将来会被复用，就已经是公共约定。 |
| “技术文档里写了，不用再维护规则文档” | 单次技术文档不能替代长期公共契约。 |
| “先把代码合进去，规则回头再补” | 规则滞后会让后续开发继续基于错误前提推进。 |

## 最终结论

`middleware-contracts` 的职责是：**在正确时机驱动读取和更新项目的中间件与跨模块统一约定文档。**

**公共规则不能只活在单次方案里，也不能只活在代码里。**
