# 测试与交付

## 概述

这个阶段的目标不是“写一份总结”，而是形成一份**开发者或测试执行者可以直接照着执行、执行完还能直接记录结果并支撑交付判断**的 `testing_delivery.md`。

本 skill 主要回答：

- 这次改动到底测什么
- 测试前提和环境准备是什么
- 具体该怎么测
- 实际结果是什么
- 还有哪些遗留问题和交付风险
- 当前项目总介绍需要同步什么

它不直接负责：

- 重新定义需求
- 补做技术方案
- 替代代码实现阶段的改造记录

这些分别属于 `requirement-analysis`、`technical-design` 和 `implementation-execution`。

**核心原则：** 测试与交付文档必须优先服务执行，不允许写成只适合汇报、不适合照着测试的空泛总结。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始前至少读取：

1. `AGENTS.md`
2. `project_info.md`
3. 当前模块当前期次目录 `feature_info.md`
4. `requirement.md`
5. `.agents/skills/test-and-delivery/testing_delivery.template.md`

按需补读：

6. `technical_design.md`
7. `implementation_report.md`
8. 实际测试结果、日志、截图、命令输出或联调记录

## 进入条件

只有满足以下条件时，才允许进入本 skill：

1. 已完成 `implementation-execution`，或当前实现已达到可验证状态
2. 当前模块目录与当前期次目录存在
3. 当前期次的需求边界已稳定，不再处于补需求阶段
4. 已有真实测试结果，或当前正准备按测试单执行测试

如果实现还没完成、测试前提还没准备好，就不要开始写完整的 `testing_delivery.md`。

## 输出

输出文件固定为：

`docs/模块开发文档/<模块名称>/<phase>/testing_delivery.md`

同时必须同步维护：

- `project_info.md`
- 当前模块当前期次目录 `feature_info.md`

`testing_delivery.md` 必须遵循：

- `.agents/skills/test-and-delivery/testing_delivery.template.md`

## 阶段一：校验测试是否就绪

在正式写文档前，先确认以下条件：

1. 本期实现范围已经明确
2. 当前期次需要覆盖的主流程和关键异常流程已知
3. 测试环境、测试数据和测试前提可描述
4. 已明确哪些结果来自真实执行，哪些仍待执行

### 阶段一输出

如果当前仍不具备完整测试条件，只允许输出：

1. 当前已具备的测试前提
2. 当前缺失的测试条件
3. 缺失条件为什么会阻塞最终交付结论

此时必须继续补测试条件，不允许伪造“已验证完成”的结论。

## 阶段二：整理可执行测试单

按模板把测试文档写成**可执行的测试单**。

最低要求必须覆盖：

1. 文档修订记录
2. 测试目标与交付范围
3. 测试环境与准备
4. 测试数据准备
5. 主流程测试用例
6. 异常与边界测试用例
7. 回归检查项
8. 执行结果记录
9. 问题与处理情况
10. 交付结论

写作规则：

- 测试步骤要能被开发者或测试执行者直接照着执行
- 预期结果要和需求边界一致
- 如果某项尚未执行，必须明确标记，不允许伪装成已完成
- 如果本次实现有显著偏差，要结合 `implementation_report.md` 说明验证重点

## 阶段三：记录真实结果

测试文档不能停留在“计划”。

至少要明确：

1. 哪些测试已执行
2. 执行结果是什么
3. 有哪些问题已修复
4. 有哪些问题仍遗留
5. 哪些风险已接受，哪些风险阻塞交付

补充要求：

- 命令、接口调用、页面操作、日志检查等证据应尽量落文字记录
- 不能用“基本正常”“看起来没问题”替代执行结果
- 若本次未覆盖某个高风险场景，必须明确写出原因

## 阶段四：更新 feature_info 和 project_info

写完 `testing_delivery.md` 后，必须同步更新：

- 当前模块当前期次目录 `feature_info.md`
- `project_info.md`

至少回填：

- 当前状态：`测试与交付中` / `待最终审核` / `可交付`
- 当前期次已产出的文档清单
- 本次模块开发完成后需要写入 `project_info.md` 的能力变化、入口变化或约定变化

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- `testing_delivery.md` 已写完
- 文档已经同时包含测试执行方式和实际结果
- `feature_info.md` 已同步更新
- `project_info.md` 已同步更新
- 已明确告知用户当前进入最终审核或交付判断阶段

## 红旗信号 - 立即停止

如果你发现自己开始做以下事情，说明已经跑偏，必须停下来：

- 只写测试计划，不写实际结果
- 用空泛结论代替可执行步骤
- 把需求或技术方案大段抄进测试文档
- 没有测试证据却宣称“已完成验证”
- 没更新 `project_info.md` 就宣称交付完成
- 把遗留风险隐藏在模糊措辞里

这些都属于**不可审核、不可执行、不可交付**的测试文档。

## 常见自我合理化

| Excuse | Reality |
|--------|---------|
| “先把测试计划写上，结果以后再补” | 没有真实结果的交付文档不能支撑放行。 |
| “这部分我手工点过，大概没问题” | 手工验证也要记录步骤和结果，不能只给印象判断。 |
| “L1 功能就不用写得太细了” | 复杂度可以影响篇幅，但不能取消可执行性和结果记录。 |
| “问题已经口头说过了，不用写进文档” | 没有落文档的问题，后续就无法追踪和审计。 |
| “project_info.md 回头再更新” | 项目现状不同步，后续 AI 和人都会基于过时信息继续工作。 |

## 最终结论

`test-and-delivery` 的职责是：**形成一份能直接执行、能真实记录结果、能支撑最终交付判断的测试与交付文档。**

**没有真实结果记录，就没有合格的交付文档。**
