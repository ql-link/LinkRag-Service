# 项目启动

## 概述

在 ToLink 项目中，任务开始时最容易犯的错误不是“不了解代码”，而是**需求还没聊清楚就开始判级、建目录、写文档**。

本 skill 的作用是：

1. 先加载项目上下文
2. 先进入需求讨论
3. 只有边界足够清楚后，才允许输出复杂度和计划

**核心原则：** 先讨论，后判级；先澄清，后建档。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始时必须先读取：

1. `AGENTS.md`
2. `project_info.md`
3. `.agents/skills/middleware-contracts/SKILL.md`（若需求涉及跨模块约定或中间件）
4. `docs/architecture/middleware_contract.md`（若需求涉及跨模块约定或中间件）
5. 当前模块期次目录 `feature_info.md`（若任务已绑定到具体目录）

## 进入条件

在需求讨论充分前，禁止做以下动作：

- 判断 `L1 / L2 / L3`
- 判断是否拆 `一期 / 二期`
- 创建模块目录或期次目录
- 生成 `feature_info.md`
- 生成 `requirement.md`
- 进入 `technical-design`
- 开始代码实现

**只有在关键边界已经足够明确后，才允许进入复杂度分析与文档计划阶段。**

## 阶段一：上下文加载

先完成以下动作：

1. 读取项目宪法与项目现状
2. 判断需求是否涉及跨模块契约或中间件
3. 如果用户已经指定某个模块或期次，确认当前目录和阶段是否一致
4. 用一句话复述当前理解的需求

### 阶段一输出

只允许输出：

1. 当前需求理解摘要
2. 当前已明确范围
3. 当前仍待澄清问题
4. 下一轮需要确认的关键维度

此阶段不允许输出复杂度建议，除非关键边界已经足够清楚。

## 阶段二：需求讨论

在这个阶段，必须围绕需求边界持续提问和澄清。

至少应覆盖以下维度中会影响方案方向的部分：

1. 需求目标
2. 本期范围与明确不做范围
3. 主业务维度
4. 参与角色
5. 主流程
6. 关键异常流程
7. 是否拆分 `一期 / 二期`
8. 各端或各系统职责边界
9. 关键状态与结果
10. 需要哪些数据对象
11. 外部依赖与中间件依赖
12. 兼容或迁移要求
13. 验收标准

### 讨论规则

- 一次只追问会影响方向的问题
- 优先追问真正阻塞后续分析的问题
- 如果某个维度还不清楚，就不要假装已经明确
- 如果用户需求天然跨阶段，要先帮助用户拆一期/二期，而不是直接吞成一个大需求

## 阶段三：复杂度与计划

只有在需求讨论充分后，才允许输出以下内容：

1. 当前需求理解摘要
2. 影响模块与中间件初判
3. 建议复杂度等级 `L1 / L2 / L3`
4. 是否建议拆分 `一期 / 二期`
5. 判级与拆分依据
6. 本次预计产出的文档清单
7. 建议创建的模块目录与当前期次目录
8. 请求用户确认等级和期次拆分后再进入下一阶段

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- 已完成需求讨论
- 关键边界已明确到足以进入需求分析
- 用户已确认复杂度等级
- 用户已确认是否拆分 `一期 / 二期`
- 已给出模块目录与期次目录建议

如果以上任一条件不满足，本 skill 不应宣称完成。

## 红旗信号 - 立即停止

如果你发现自己开始做以下事情，说明已经跑偏，必须停下来：

- “先按 L2 处理吧，后面再说”
- “先建目录，边写边补”
- “先写个 requirement.md 草稿”
- “先默认一期不拆”
- “这应该就是数据集维度”
- “Java 端大概负责这些，后面再确认 Python”

这些都属于**在边界未确认前提前下结论**。

## 常见自我合理化

| Excuse | Reality |
|--------|---------|
| “需求差不多清楚了，先判级再说” | 差不多不等于足够明确。先补齐关键边界。 |
| “先建目录不会有问题” | 目录一建，后面就容易把错误边界写进文档。 |
| “先写草稿再问细节更快” | 草稿会反向固化错误假设，后面更难改。 |
| “这个需求看起来很常见” | 常见需求也可能因为主维度、职责边界不同而完全不同。 |
| “先默认不拆一期二期” | 分期本身就是需求边界的一部分，必须讨论。 |

## 最终结论

在 ToLink 项目里，`project-bootstrap` 不是“快速判级器”，而是**需求讨论门禁**。

**没有充分讨论，就没有复杂度分析；没有边界确认，就没有正式文档。**
