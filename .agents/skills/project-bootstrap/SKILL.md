# Project Bootstrap

## 目的

在任务正式进入需求分析、技术设计或实现之前，先统一完成上下文加载与需求讨论；只有边界足够明确后，才进入复杂度判级。

## 必读文件

1. `AGENTS.md`
2. `project_info.md`
3. `docs/architecture/middleware_contract.md`（若需求涉及跨模块约定或中间件）

## 第一阶段输出要求：需求讨论

必须先输出以下内容：

1. 当前需求理解摘要
2. 当前已明确范围
3. 当前仍待澄清问题
4. 下一轮需要与用户确认的关键维度

在这一阶段，不得直接输出复杂度等级、期次拆分建议或正式文档计划，除非关键边界已经足够明确。

## 需求讨论阶段必澄清维度

至少应覆盖以下维度中会影响方案方向的部分：

1. 需求目标
2. 本期范围与明确不做范围
3. 主业务维度
4. 参与角色
5. 主流程
6. 关键异常流程
7. 是否拆分 `一期 / 二期`
8. 各端或各系统职责边界
9. 关键状态与结果
10. 需要哪些数据对象
11. 外部依赖与中间件依赖
12. 兼容或迁移要求
13. 验收标准

## 第二阶段输出要求：复杂度与计划

只有在需求讨论充分后，才允许输出以下内容：

必须输出以下内容：

1. 当前需求理解摘要
2. 影响模块与中间件初判
3. 建议复杂度等级 `L1/L2/L3`
4. 是否建议拆分 `一期 / 二期`
5. 判级与拆分依据
6. 本次预计产出的文档清单
7. 建议创建的模块目录与当前期次目录
8. 请求用户确认等级和期次拆分后再进入下一阶段

## 强制约束

- 在需求讨论未充分前，不允许进入复杂度分析
- 未确认复杂度等级前，不允许进入正式需求文档编写
- 未确认是否拆分 `一期 / 二期` 前，不允许直接创建最终文档目录
- 若任务已绑定某个模块期次目录，继续前必须读取该目录 `feature_info.md`
