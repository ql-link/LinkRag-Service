# Requirement Analysis

## 1. 目的

本 skill 用于把已经过初步澄清的功能需求，整理成结构化的 `requirement.md`。

它只回答：

- 为什么做
- 做什么
- 给谁做
- 边界是什么
- 流程是什么
- 验收标准是什么

它不回答：

- Controller 怎么写
- Service 怎么拆
- 表结构怎么改
- Redis / MQ / OSS 怎么接

这些内容属于后续 `technical-design`。

## 2. 使用前提

只有满足以下条件时，才允许使用本 skill：

1. 已完成 `project-bootstrap`
2. 用户已确认复杂度等级 `L1 / L2 / L3`
3. 用户已确认是否拆分 `一期 / 二期`
4. 已创建或确认当前模块目录与当前期次目录
5. 已确认当前阶段是“需求分析”，不是直接实现或技术设计

如果复杂度等级或期次拆分未确认，必须先停下来让用户确认，不能提前写 `requirement.md`。

## 3. 必读文件

执行本 skill 时，至少读取以下文件：

1. `AGENTS.md`
2. `project_info.md`
3. 当前模块当前期次目录 `feature_info.md`
4. `docs/module-development-files/README.md`
5. `.agents/skills/requirement-analysis/PRD.template.md`

按需补读：

6. 同业务域历史模块目录中的 `requirement.md`
7. 同业务域历史模块目录中的 `technical_design.md`
8. 相关业务背景文档
9. `docs/architecture/middleware_contract.md`（若需求边界明显涉及 MySQL / Redis / MQ / OSS）

注意：

- 这里读取 `middleware_contract.md` 只是为了识别边界与术语，不是为了写技术方案
- 不应在本阶段因为看到了组件能力就直接进入实现设计

## 4. 输出位置

输出文件固定为：

`docs/module-development-files/<domain>-<module-name>/<phase>/requirement.md`

注意：

- 输出文件名仍使用 `requirement.md`
- 但文档内容结构必须遵循 `.agents/skills/requirement-analysis/PRD.template.md`

若目录中已有旧版 `requirement.md`：

- 先读旧版
- 判断是覆盖、修订还是增量更新
- 不允许无说明地重写并覆盖关键结论

## 5. 同时需要维护的文件

除 `requirement.md` 外，本 skill 还应同步维护当前模块当前期次目录的 `feature_info.md`。

至少应回填：

- 当前状态：`需求分析中` / `需求待审核`
- 复杂度等级
- 本次需求摘要
- 本次实际产出的文档清单
- 推荐阅读顺序

## 6. 输出内容要求

`requirement.md` 必须优先遵循 `.agents/skills/requirement-analysis/PRD.template.md` 的章节结构。

最低要求包括：

1. 文档修订记录
2. 需求背景与业务目标
3. 核心架构与业务流程
4. 功能规格与交互逻辑
5. 数据契约与存储约束
6. 异常处理与非功能性需求
7. 遗留问题与依赖项

如果需求较轻，可以适度精简内容密度，但不要擅自删除一级章节。

写作要求：

- 用需求语言，不用实现语言
- 范围要可验证，不写空泛词
- 主流程和异常流程要能指导后续设计
- 验收标准必须能被测试或人工审核验证
- 时序图、状态机、数据变更表等内容按 PRD 模板填写

## 7. 需求分析的工作步骤

### 步骤 1：确认功能目标

先用一句话重述本次功能目标，确保目标不是技术方案表述。

在进入后续步骤前，还必须先判断：

- 当前模块是否应拆为 `一期 / 二期`
- 当前正在分析的是哪一期
- 本期边界是否可以独立交付

正确示例：

- “支持用户按数据集上传原始知识文件并查看上传结果”

错误示例：

- “新增 KnowledgeFileService 和 OSS 上传接口”

### 步骤 2：拆出角色与使用场景

至少识别：

- 谁发起操作
- 谁查看结果
- 是否有管理员 / 系统内部角色 / 外部系统角色

### 步骤 3：明确边界

必须明确：

- 本次必须实现
- 可选项
- 明确不做

如果边界不清楚，本 skill 的首要任务不是写文档，而是先把边界问清楚。

### 步骤 4：梳理流程

至少写清：

- 主成功路径
- 关键失败路径
- 对用户可见的状态或结果

### 步骤 5：补充数据、权限与验收

说明：

- 需要保存什么数据
- 哪些数据有权限边界
- 成功、失败、拒绝分别如何判定

## 8. 提问门禁

以下问题如果不清楚，必须先向用户提问，不能直接产出最终 `requirement.md`：

- 功能范围是否不明确
- 谁能使用、谁不能使用不明确
- 主流程或异常流程不明确
- 数据是否需要保存、保留多久、谁可见不明确
- 是否涉及外部系统 / 回调 / 异步协作不明确
- 验收标准不明确
- 这些不确定项会直接影响后续表结构、接口、权限或技术方案

提问规则：

- 只问能阻塞文档的问题
- 问题要短，不要一口气展开成长问卷
- 问题应说明“为什么这件事会影响需求边界”

## 9. 不允许写进需求文档的内容

以下内容不要提前写进 `requirement.md`：

- 具体 API 路径
- Controller / Service / Mapper 类名
- 表名、字段名、索引名
- Redis key 设计
- MQ topic / group 设计
- OSS object key 实现方案
- 事务、幂等、重试设计

如果用户已经提到这些内容，可以在“约束”或“背景”中点到为止，但不要展开成实现设计。

例外：

- 若 PRD 模板中的“数据库组件与表结构变更”章节需要提前表达需求级约束，可以写“需要新增/调整哪类存储对象或中间件能力”
- 但不要在本阶段把它写成最终技术设计或具体实现方案

## 10. 输出质量标准

一份合格的 `requirement.md` 至少要做到：

- 另一个工程师不看聊天记录，也能知道这次要做什么
- 另一个 AI 不看技术代码，也能知道后续技术设计要解决哪些问题
- 不会把需求文档写成技术设计文档
- 不会留下影响实现方向的大坑但假装已经写完

应避免的表达：

- “完善功能”
- “优化体验”
- “增加相关逻辑”
- “按需处理”
- “后续补充”

除非这些内容被明确放进“待确认问题”。

## 11. 写完后的停点

本 skill 完成后必须停在“人工审核需求文档”这个节点。

完成动作包括：

1. 写好 `requirement.md`
2. 更新 `feature_info.md`
3. 明确告知用户当前已进入“需求待审核”
4. 等待用户确认，不允许直接跳到 `technical-design`

## 12. 推荐输出摘要

本 skill 完成后，对用户的摘要建议至少包含：

- 本次需求文档路径
- 需求范围摘要
- 当前待确认问题（如果有）
- 下一步需要用户做什么：审核需求文档

## 13. 与其他 skill 的衔接

- 进入前：应先完成 `project-bootstrap`
- 结束后：等待人工审核
- 审核通过后：再进入 `technical-design`

禁止行为：

- 未经审核直接进入 `implementation-execution`
- 在本阶段代替 `technical-design` 提前产出实现方案
