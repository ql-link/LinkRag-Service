# 需求分析

## 概述

这个阶段的目标不是“把聊天记录整理一下”，而是把已经讨论清楚的需求，固化成**可审核、可分期、可指导后续技术设计**的 `requirement.md`。

本 skill 只回答这些问题：

- 为什么做
- 做什么
- 给谁做
- 边界是什么
- 主流程和异常流程是什么
- 验收标准是什么

它不回答：

- Controller 怎么写
- Service 怎么拆
- 表字段怎么定
- Redis / MQ / OSS 怎么接

这些属于后续 `technical-design`。

**核心原则：** 需求文档锁定边界，不提前代替技术方案。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始前至少读取：

1. `AGENTS.md`
2. `project_info.md`
3. 当前模块当前期次目录 `feature_info.md`
4. `docs/模块开发文档/README.md`
5. `.agents/skills/requirement-analysis/PRD.template.md`

按需补读：

6. 同业务域历史模块目录中的 `requirement.md`
7. 同业务域历史模块目录中的 `technical_design.md`
8. 相关业务背景文档
9. `.agents/skills/middleware-contracts/SKILL.md`（若需求边界明显涉及 MySQL / Redis / MQ / OSS）
10. `docs/组件和数据库约定/middleware_contract.md`（若需求边界明显涉及 MySQL / Redis / MQ / OSS）

## 进入条件

只有满足以下条件时，才允许进入本 skill：

1. 已完成 `project-bootstrap`
2. 需求讨论已充分，关键边界已明确
3. 用户已确认复杂度等级 `L1 / L2 / L3`
4. 用户已确认是否拆分 `一期 / 二期`
5. 已创建或确认当前模块目录与当前期次目录
6. 已确认当前阶段是“需求分析”

如果以上任一条件不满足，就不要开始写 `requirement.md`。

## 输出

输出文件固定为：

`docs/模块开发文档/<模块名称>/<phase>/requirement.md`

同时必须同步维护：

- 当前模块当前期次目录 `feature_info.md`

`requirement.md` 必须遵循：

- `.agents/skills/requirement-analysis/PRD.template.md`

## 阶段一：校验范围是否就绪

在正式写文档前，先做最后一次需求边界校验：

1. 本期范围是否明确
2. 本期不做是否明确
3. 当前期次是否明确
4. 主业务维度是否明确
5. 主要角色与职责边界是否明确
6. 主流程与关键异常流程是否明确
7. 验收标准是否明确

### 阶段一输出

如果仍有关键空白，只允许输出：

1. 当前已明确范围
2. 当前仍待确认问题
3. 为什么这些问题会阻塞 `requirement.md`

此时必须继续提问，不允许写正式文档。

## 阶段二：编写需求文档

当边界已足够明确后，按模板写 `requirement.md`。

最低要求必须覆盖：

1. 文档修订记录
2. 业务层
3. 架构约束层
4. 技术边界层
5. 风险、依赖与待确认问题

写作规则：

- 用需求语言，不用实现语言
- 范围要可验证，不写空泛词
- 主流程和异常流程必须能指导后续设计
- 验收标准必须能被测试或人工审核验证
- PRD 中的数据模型采用对象级模型，不提前写成字段级模型
- 可以提前锁定关键技术约束，但不要直接展开成最终技术实现

## 阶段三：更新 feature_info

写完 `requirement.md` 后，必须同步更新 `feature_info.md`，至少回填：

- 当前状态：`需求分析中` / `需求待审核`
- 复杂度等级
- 本次需求摘要
- 本次实际产出的文档清单
- 推荐阅读顺序

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- `requirement.md` 已写完
- `feature_info.md` 已同步更新
- 当前文档已进入“需求待审核”状态
- 已明确告知用户下一步是审核需求文档

## 红旗信号 - 立即停止

如果你发现自己开始做以下事情，说明已经跑偏，必须停下来：

- 直接写 API 路径
- 直接写类名、包名、实现分层
- 直接定表字段、索引、SQL
- 直接设计 Redis key / MQ topic / object key
- 一边写 PRD，一边替技术方案做实现决策

这些都属于**需求文档滑向技术文档**。

## 常见自我合理化

| Excuse | Reality |
|--------|---------|
| “先把接口也写进去，后面省事” | 接口细节属于技术方案，不属于需求边界。 |
| “先把表字段定了，AI 后面更好写” | 字段级模型过早锁死实现，会污染需求分析。 |
| “这部分技术约束已经很明显了，直接写实现吧” | PRD 可以锁原则，但不能代替技术设计。 |
| “先把 requirement.md 写出来再回头问” | 文档会固化错误假设，后面更难改。 |
| “这就是一个简单功能，不需要完整 PRD” | 只要进入需求分析阶段，就要形成可审核文档。 |

## 最终结论

`requirement-analysis` 的职责是：**把已讨论清楚的需求，固化成可审核的边界文档。**

**它不是技术设计，也不是实现草稿。**
