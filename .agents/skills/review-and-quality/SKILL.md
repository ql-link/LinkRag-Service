# 评审与质量

## 概述

本 skill 的目标不是“写个总结”，而是用正式评审视角找出当前产物里的真实问题，尤其是：

- 需求边界是否错了
- 技术方案是否脱离代码现实
- 实现是否和方案不一致
- 是否存在行为回归、隐藏风险或漏测
- 文档、代码、测试三者是否一致

它主要服务于：

- `requirement-analysis` 之后的需求评审
- `technical-design` 之后的技术评审
- `implementation-execution` 之后的代码评审
- `test-and-delivery` 之后的交付评审

同时它承担**完成前验证**职责：

- 在准备宣称“已完成 / 已修复 / 已通过 / 可提交”之前
- 要求用最新验证证据支撑结论

**核心原则：** 评审以发现问题和风险为主，不以“给出好看总结”为主。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始前至少读取：

1. `AGENTS.md`
2. `project_info.md`
3. 当前模块当前期次目录 `feature_info.md`

按评审对象补读：

4. `requirement.md`
5. `technical_design.md`
6. `implementation_report.md`
7. `testing_delivery.md`
8. 相关代码改动
9. 相关公共契约或组件说明文档

## 进入条件

在以下场景中适用本 skill：

1. 需求文档写完，准备人工审核
2. 技术方案写完，准备人工审核
3. 代码实现完成，准备检查风险和回归
4. 测试与交付文档完成，准备做最终放行判断

如果当前产物还处于明显未完成状态，不要假装进入评审，而应该先回到对应主流程继续补齐。

## 输出

输出应优先是**问题列表**，而不是概述。

默认输出顺序：

1. Findings：按严重程度列出问题
2. Open Questions / Assumptions：当前仍存在的不确定项
3. Change Summary：只有在必要时才补简短总结

问题格式建议：

- 严重级别
- 文件或文档位置
- 问题描述
- 为什么这是问题
- 可能影响

## 阶段一：确定评审对象和标准

先明确当前在评审哪一层：

1. 需求评审
2. 技术评审
3. 代码评审
4. 交付评审

然后明确对应标准：

- 需求评审：看边界、分期、职责、验收标准
- 技术评审：看代码依据、方案合理性、异常和中间件边界
- 代码评审：看行为正确性、回归风险、注释、契约一致性
- 交付评审：看测试覆盖、遗留风险、交付是否成立

## 阶段二：先找问题，不先写总结

评审时优先查找：

- 是否有逻辑错误
- 是否有边界遗漏
- 是否有行为回归风险
- 是否有实现与文档不一致
- 是否有缺失测试或无证据结论
- 是否有违反公共契约的地方

不要先写“整体不错”“大体可行”这种低价值结论。

## 阶段三：按评审类型给出重点

### 需求评审重点

- 本期范围和不做范围是否清楚
- 是否错误混入了技术实现细节
- 一期/二期拆分是否合理
- 验收标准是否可验证

### 技术评审重点

- 是否引用了真实代码和真实组件
- 是否空想模块边界
- 是否明确了异常类、错误码、中间件接入方式
- 是否和公共契约冲突

### 代码评审重点

- 是否按已确认边界实现
- 是否有未授权扩范围
- 关键状态流转和复杂逻辑是否有注释
- 是否存在明显 bug、风险或回归

### 交付评审重点

- 测试是否覆盖主流程和关键异常
- 是否真的记录了实际结果
- 是否还有阻塞交付的风险
- `project_info.md` 是否同步

## 阶段四：完成前验证

在准备给出“已完成 / 已修复 / 已通过 / 可提交”结论前，必须额外完成：

1. 明确你准备声称什么
2. 明确哪条命令或哪组检查能证明它
3. 运行最新验证
4. 阅读完整输出
5. 只在证据成立后再给结论

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- 已明确当前评审对象
- 已优先输出问题列表
- 问题按严重程度排列
- 已指出关键风险、缺口或阻塞项

## 红旗信号 - 立即停止

- 先写总结，后找问题
- 因为是自己写的内容就放松标准
- 只看格式，不看行为和边界
- 只说“建议优化”，不指出真实风险
- 没读相关文档或代码就直接给通过意见
- 没有最新验证证据就声称“完成”“通过”“可提交”
- 用“应该”“大概”“看起来”代替验证结论

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| “这次只是轻量评审，简单看看就行” | 越轻量越容易放过真实问题。 |
| “先写整体印象，再补细节” | 评审价值在问题，不在印象。 |
| “这个风险实现时再看” | 如果现在已经能看出风险，就应该现在指出。 |
| “测试大概够了” | 没证据的“大概”不能支撑放行。 |
| “文档和代码差一点没关系” | 文档与代码不一致会直接误导后续开发和测试。 |
| “我刚才应该已经验证过了” | 只要有新改动，旧验证结果就可能过期。 |
| “改动很小，不用再验证” | 小改动一样可能导致错误完成。 |

## 最终结论

`review-and-quality` 的职责是：**在阶段放行前，以问题和风险为中心做正式评审，并在宣称完成前要求最新验证证据。**

**没有问题清单的评审，或没有证据的完成结论，都不算真正过关。**
