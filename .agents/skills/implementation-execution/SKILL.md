# 代码实现

## 概述

这个阶段的目标不是“尽快把代码写完”，而是**按已确认边界落地实现，并在实现偏离方案时留下清晰记录**。

本 skill 主要负责：

- 根据已审核文档落地代码
- 将当前期次目标拆成小而可执行的实现任务
- 控制实现不要偏离需求和技术方案
- 在需要时产出 `implementation_report.md`

**核心原则：** 代码实现必须服从已确认边界；实现偏差必须显式留痕。

**违反流程字面要求，就是违反流程本意。**

## 所需上下文

开始前至少读取：

1. `AGENTS.md`
2. `project_info.md`
3. 当前模块当前期次目录 `feature_info.md`
4. 当前模块当前期次目录 `requirement.md`
5. 当前模块当前期次目录 `technical_design.md`（若存在）
6. 对应组件说明文档（若涉及）
7. 涉及模块的真实代码

## 进入条件

只有满足以下条件时，才允许进入本 skill：

1. `requirement.md` 已审核通过
2. 若存在 `technical_design.md`，则其也已审核通过
3. 当前边界已经稳定，不再处于需求或技术讨论阶段

如果这些条件不满足，就不要开始写代码。

## 输出

代码改动直接落在仓库中。

若需要改造报告，输出位置固定为：

`docs/module-development-files/<domain>-<module-name>/<phase>/implementation_report.md`

改造报告模板固定为：

`.agents/skills/implementation-execution/implementation_report.template.md`

## 阶段一：校验实现是否就绪

在动手前先确认：

1. 当前实现边界是否与 `requirement.md` 一致
2. 当前实现依据是否与 `technical_design.md` 一致（若存在）
3. 相关模块和组件代码是否已读
4. 当前实现是否可能触发复杂度升级或范围变化

### 阶段一输出

如果仍有关键空白，只允许输出：

1. 当前已确认的实现前提
2. 当前仍待确认的问题
3. 为什么这些问题会阻塞代码实现

此时必须停下来，不允许边写边猜。

## 阶段二：按确认边界实现

按已确认文档落代码。

在正式编码前，应先把当前期次拆成小任务，再逐项实现。

任务拆分要求：

- 每个任务都应边界清楚
- 每个任务都应有完成标志
- 先主链路，后增强项
- 先基础模块，后联动模块
- 跨模块或跨中间件任务要明确依赖顺序

实施规则：

- 以 `requirement.md` 为边界
- 以 `technical_design.md` 为实现依据
- 先复用已有模块和组件，再考虑改 framework
- 不允许跳过已确认边界私自扩展范围
- 异常类和错误码的代码实现必须遵循仓库现有异常体系与编码规范，优先复用现有 `BusinessException`、错误码定义和统一响应规则
- 涉及 Redis、OSS、MQ 等中间件时，编码前必须先读对应中间件组件说明文档，不允许只凭经验直接接入或扩 framework
- 一次优先完成一个小任务，不要把多个高风险改动捆在一个实现批次里
- 每完成一个任务，都应做一次局部验证，再进入下一个任务

## 阶段三：补齐注释与注解契约

在实现过程中，关键逻辑必须补充注释；对外 DTO、Controller 等接口层对象必须补齐必要注解。

核心原则：

- 注释解释“为什么、边界、风险、协作契约”，不要重复代码已经清楚表达的“做什么”
- 注解表达“机器可读的接口、校验、权限、事务、序列化规则”
- 注释和注解必须随着代码同步更新，过期注释比没有注释更危险

至少在以下位置补注释：

- 复杂业务判断
- 关键状态流转
- 跨组件调用链
- 不易直观看懂的设计意图
- 幂等、补偿、重试、超时、失败兜底等异常处理逻辑
- Redis、MQ、OSS、DB 事务等中间件边界和一致性边界
- 内部接口、鉴权、脱敏、私有对象 key 等安全边界

注释规则：

- 注释应解释“为什么这样做”
- 注释应简洁、解释性强
- 不要写成机械式逐行翻译

企业级 Java 注释要求：

- Controller 类必须有 JavaDoc，说明接口组职责、适用对象和关键边界
- Controller 每个 public 方法必须有 JavaDoc，说明接口职责、关键参数、返回结果和重要副作用
- Service 的 public 方法、复杂 private/helper 方法必须说明业务职责和状态/事务边界
- 涉及状态流转时，必须说明状态由谁创建、谁推进、谁终结，以及失败时如何处理
- 涉及中间件时，必须说明 key/topic/bucket/path、幂等依据、重试或补偿策略
- 不要添加“设置字段”“调用 service”“返回结果”这类无信息量注释

企业级 Java 注解要求：

- DTO、Request、Response 字段使用 `@Schema` 描述业务含义、示例值、枚举值和是否隐藏
- 内部字段、敏感字段或不应暴露给前端的字段必须使用合适的序列化控制，例如 `@JsonIgnore`，必要时配合 `@Schema(hidden = true)`
- Controller 方法优先使用 `@Operation`、`@Parameter` 补充 OpenAPI 信息；JavaDoc 负责说明职责和边界
- 请求入参必须使用 `@Valid`/`@Validated` 和 `@NotNull`、`@NotBlank`、`@Size`、`@Min`、`@Max`、`@Pattern` 等校验注解前置基础规则
- 权限注解必须靠近接口入口，例如 `@SaCheckLogin`、`@SaCheckRole`，不要只依赖 Service 层兜底
- `@Transactional` 只放在真正需要事务边界的方法上，避免把 OSS、MQ、远程调用等慢 IO 包进数据库事务
- Lombok 注解按对象职责谨慎使用：DTO 可使用 `@Data`，实体、领域对象或复杂对象避免无脑生成 setter/equals

## 阶段四：识别并记录偏差

在编码过程中持续判断：

1. 是否与技术文档一致
2. 是否新增了未预期的模块改动
3. 是否改变了原约定中的边界
4. 是否对后续测试、交付、审查产生额外影响

以下任一情况成立时，必须写 `implementation_report.md`：

1. 当前期次功能等级为 `L3`
2. 实际实现明显偏离 `technical_design.md`
3. 改动跨多个模块、多个中间件或多个关键链路
4. 存在需要向后续测试、交付、审查特别说明的实现差异

以下情况通常可以不单独写：

- `L1` 小改动
- `L2` 且实现与技术方案基本一致
- 影响面小，且后续 `testing_delivery.md` 足以说明交付结果

## 阶段五：更新 feature_info 并交接

实现完成后，至少完成以下动作：

1. 代码实现完成
2. 必要时写好 `implementation_report.md`
3. 更新 `feature_info.md`
4. 进入 `test-and-delivery` 阶段

## 完成标准

只有满足以下条件，才算本 skill 执行完成：

- 代码改动已完成
- 关键逻辑已补注释
- 若触发条件成立，`implementation_report.md` 已写完
- `feature_info.md` 已同步更新
- 已明确进入 `test-and-delivery`
- 当前实现过程按任务逐步推进，而不是一次性大爆炸改动

## 红旗信号 - 立即停止

如果你发现自己开始做以下事情，说明已经跑偏，必须停下来：

- 需求边界还没确认就开始改代码
- 技术方案没过审就直接实现
- 发现偏差但不记录
- 一边实现一边偷偷扩需求
- 关键状态流转没有任何注释
- 修改了公共契约却不回写文档

这些都属于**实现先行、治理缺位**。

## 常见自我合理化

| Excuse | Reality |
|--------|---------|
| “先写完再补文档” | 改造报告和偏差记录一旦滞后，很快就会失真。 |
| “这点偏差很小，不用记” | 小偏差如果影响测试、交付或审查，就必须留痕。 |
| “注释后面再补” | 关键逻辑不及时注释，后面最容易漏掉设计意图。 |
| “顺手把这个也改了” | 这就是未授权扩范围。先停下来确认。 |
| “先改 framework 比较省事” | 业务问题优先在业务层解决，framework 改动需要更高门槛。 |
| “先一口气做完再统一验证” | 大爆炸实现最容易隐藏回归和偏差。 |

## 最终结论

`implementation-execution` 的职责是：**按确认后的边界落地实现，并把所有关键偏差和实现意图留在仓库里。**

**没有边界约束的实现，不是高效，是失控。**
