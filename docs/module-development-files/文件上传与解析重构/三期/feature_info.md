# 功能信息卡

## 1. 基础信息

- 模块名称：文件上传与解析重构
- 当前期次：三期
- 业务域：storage
- 当前状态：三期范围待审核
- 复杂度等级：L3
- 当前分支：refactor/update-file-upload-parse

## 2. 功能摘要

- 背景：当前重构一期已完成上传链路和解析文件聚合记录初始化，二期已完成解析提交、同步 MQ 投递失败回滚、Python 回调 + SSE 单通道和数据库终态兜底。
- 目标：承接老模块三期中仍未落地、且与当前重构链路仍然直接相关的能力，继续增强多实例可用性，并补齐删除/版本/下游衔接等扩展能力。
- 本期定位：
  - 不是继续重复一期/二期已实现内容。
  - 主要承接“一致性增强 + 功能补全 + 历史兼容链路清理”。

## 3. 往期三期对比结论

| 3 | MQ 投递失败即时失败化 | 二期已改为“更新数据库后同步发 MQ，发送失败直接抛错并回滚事务，不保留补偿等待态” | 不再进入三期 |

### 3.2 老模块三期中仍未实现、需要延续到当前重构三期的能力

| 4 | 原文件软删除与解析产物删除 | 当前仍是原文件对象 + DB 记录直接删除，未形成“原文件隐藏、历史保留、解析产物清理”完整策略 | 是 |
| 5 | 解析产物版本管理 | 当前只保留“最新任务/最新结果”口径，未支持多版本成功产物管理 | 是 |

| :--- | :--- | :--- |
| 旧 `parse_result` 兼容链路清理 | 旧结果 MQ 兼容代码仍保留 | 主链路已改为 Python 写库 + 内部回调，旧链路继续保留会增加误用风险 |
| 解析日志历史查询与增长治理 | 二期只完成主链路闭环，尚未补历史分页/索引/清理策略 | 日志会持续增长，后续可观测性与查询性能需要正式设计 |

## 5. 三期目标说明

### 5.1 高优先级目标

#### 目标 1：SSE 多实例跨节点进度分发

- 二期已明确不解决多实例广播一致性。
- 三期应完成：
  - 定义 Python 回调事件在多节点 Java 间的分发方案；
  - 保证“回调落到 A 节点、前端订阅在 B 节点”时仍可实时收到进度；
  - 保留数据库终态查库兜底，不以高频轮询替代事件广播。

### 5.2 中优先级目标

#### 目标 2：原文件删除语义升级

- 当前实现是原文件对象删除成功后直接删 DB 记录，适合一期/二期基础链路，但还没有形成解析侧完整删除策略。
- 三期需要明确：
  - 是否改为原文件软删除；
  - 前端列表如何隐藏已删除原文件；
  - 解析日志历史是否保留；
  - 已生成的 Markdown 解析产物由谁删、何时删、失败如何补偿。

#### 目标 3：解析产物版本管理

- 当前只保留最新任务指针和最新结果展示能力。
- 三期可补：
  - 多次成功解析版本保留；
  - 当前生效版本标识；
  - 前端查看历史版本和切换生效版本的规则。

#### 目标 4：MQ 投递可靠性增强

- 当前二期已经满足“同步失败即回滚”的业务要求，但没有进一步增强 DB 与 MQ 之间的可靠一致性。
- 三期若需要提高可靠性，可独立评估：
  - Outbox；
  - 本地消息表；
  - 统一投递重放与幂等策略。

#### 目标 5：旧兼容链路与兼容空壳清理

- 当前仓库仍保留：
  - 旧 `parse_result` 兼容消费链路；
  - 旧补偿调度空壳；
  - 部分旧实现空壳和误导性注释口径。
- 三期可在不影响联调的前提下完成清理，降低误读和维护成本。

#### 目标 6：解析日志查询与归档治理

- 二期 requirement 已明确“解析日志会持续增长”，但本期只做了主链路，没有补历史能力。
- 三期建议正式设计：
  - 历史解析日志分页查询；
  - 日志检索字段和索引；
  - 保留时长、归档和清理规则；
  - 面向运营/排障的查询视图。

### 5.3 低优先级目标

#### 目标 7：解析后下游知识处理衔接

- 当前重构链路止步于“解析成功并可查询结果”。
- 三期之后若业务优先级提升，可继续进入：
  - 向量化；
  - 检索；
  - 问答消费；
  - 与数据集知识生效策略联动。

## 6. 本期不纳入三期的事项

- 不再把“知识文件配置表迁移到 YAML”列入三期，因为上传主链路已完成切换。
- 不再把“MQ 投递失败即时失败化”列入三期，因为二期已按新口径实现。
- 不把二期 `testing_delivery.md` 补写本身当作三期功能目标，它属于二期测试交付收尾事项，不是新的业务能力。
