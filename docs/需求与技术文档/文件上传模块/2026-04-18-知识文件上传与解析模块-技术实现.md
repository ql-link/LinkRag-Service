## 2. 需求摘要

### 2.1 目标

- 支持登录用户在指定对话会话空间上传知识文件，由 Java 端保存原文件并写入原文件表。
- 支持用户后续创建解析任务，Java 端通过 MQ 通知 Python/RAG 端异步解析。
- 支持管理员查看并修改知识文件上传配置，至少包括单文件大小上限和允许格式白名单。
- 配置修改后对后续上传请求立即生效，不要求重启服务。

### 2.2 范围

必须实现：

- 单文件上传，绑定现有 `chat_conversation.id`。
- 原文件保存到私有 OSS。
- Java 端维护 `document_original_file` 原文件记录。
- Java 端维护管理员可修改的知识文件配置。
- 上传校验优先读取数据库中的当前生效配置；数据库无记录时回退 YAML 默认值。
- Java 端通过 `tolink.rag.parse_task` 投递解析任务。
- Java 端消费 `tolink.rag.parse_result` 回写解析状态。
- 管理端提供配置查询、配置修改接口。

暂不实现：

- 多文件批量上传。
- 配置历史版本管理和一键回滚。
- 管理端修改 `internal-base-url`、`service-token`、MQ 名称等服务内部配置。
- 物理删除 OSS 文件。

### 3.3 关键现状

- 当前上传限制全部来自 `KnowledgeFileProperties`，属于静态 YAML 配置。
- 当前项目没有可直接复用的“系统配置表”。
- 当前管理员接口统一放在 `/api/v1/admin` 下，由 `AdminController` 汇总。
- 当前 Redis 组件已经提供双删缓存服务，适合给知识文件配置增加缓存键。

### 3.4 设计约束

- `internal-base-url`、`service-token`、MQ vendor 和 topic 名称仍属于部署级配置，继续保留在 YAML。
- 管理员动态可修改的范围仅限知识文件上传规则，不扩展到 MQ/内部服务参数。
- 由于没有现成系统配置表，本次新增独立配置表，避免把知识文件规则继续塞回 `service` 级静态配置。

## 4. 总体设计

### 4.1 设计思路

将知识文件配置拆成两层：

1. 静态基础配置：保留在 `KnowledgeFileProperties`
   - `internalBaseUrl`
   - `serviceToken`
   - MQ 相关名称
   - 默认 `allowedSuffixes`
   - 默认 `maxSizeBytes`

2. 动态业务配置：新增数据库表 `knowledge_file_config`
   - `max_size_bytes`
   - `allowed_suffixes`
   - `updated_by`
   - `updated_at`

上传校验时优先读取数据库配置；如果数据库无配置记录，则回退到 `KnowledgeFileProperties` 默认值。

### 4.2 运行链路

上传链路：

```text
KnowledgeFileController
 -> KnowledgeFileService
 -> KnowledgeFileRuntimeConfigService
 -> ChatConversationMapper
 -> KnowledgeOriginalFileMapper
 -> IOssService
```

管理员配置链路：

```text
AdminController
 -> AdminKnowledgeFileConfigService
 -> KnowledgeFileConfigMapper
 -> DoubleDeleteCacheService(可选)
```

解析链路：

```text
KnowledgeFileService / KnowledgeParseTaskService
 -> MQSend
 -> tolink.rag.parse_task
 -> Python/RAG
 -> tolink.rag.parse_result
 -> Java MQ Receiver
```

## 6. 数据模型设计

### 6.1 表结构变更

| 表 | 变更类型 | 说明 |
| --- | --- | --- |
| `knowledge_file_config` | 新增 | 管理员可维护的知识文件上传配置 |
| `document_original_file` | 已被新方案替代 | 当前已改为按 `dataset_id` 归属，采用物理删除 |
| `document_parse_task` | 已废弃 | 当前解析任务仅通过 MQ 投递，不再建本地任务表 |

### 6.2 配置表设计

```sql
CREATE TABLE `knowledge_file_config` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '配置记录ID',
  `max_size_bytes` BIGINT UNSIGNED NOT NULL COMMENT '单文件大小上限，单位字节',
  `allowed_suffixes` JSON NOT NULL COMMENT '允许上传后缀白名单，JSON数组',
  `updated_by` BIGINT UNSIGNED NOT NULL COMMENT '最后修改管理员ID',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='知识文件上传配置表';
```

设计说明：

- 首期只维护一条“当前生效配置”。
- 查询时按 `id DESC LIMIT 1` 或固定主键方式读取。
- 初始化 SQL 插入一条默认配置，值来源于 YAML 默认值。

## 7. 服务与实现设计

### 7.1 Service 设计

```java
public interface AdminKnowledgeFileConfigService {
    KnowledgeFileConfigDTO getCurrentConfig();
    void updateConfig(Long adminUserId, UpdateKnowledgeFileConfigRequest request);
}
```

```java
public interface KnowledgeFileRuntimeConfigService {
    KnowledgeFileRuntimeConfig getCurrent();
}
```

### 7.2 运行时配置读取规则

`KnowledgeFileRuntimeConfigService.getCurrent()`：

1. 先查缓存 `knowledge:file:config:current`。
2. 缓存未命中时查 `knowledge_file_config` 表最新记录。
3. 若数据库无记录，则回退 `KnowledgeFileProperties` 默认值。
4. 返回统一运行时对象：
   - `maxSizeBytes`
   - `allowedSuffixes`

### 7.3 管理员配置修改流程

1. 管理员调用 `PATCH /api/v1/admin/knowledge-file-config`。
2. `AdminController` 复用 `@SaCheckRole("ADMIN")` 做权限校验。
3. `AdminKnowledgeFileConfigService` 校验参数。
4. 将 `allowedSuffixes` 标准化后写入数据库。
5. 记录 `updated_by = 当前管理员ID`。
6. 驱逐 Redis 缓存键 `knowledge:file:config:current`。
7. 后续上传请求自动读取新配置。

### 7.4 上传主流程调整

当前 `KnowledgeFileServiceImpl.validateFile` 需要从直接读 `KnowledgeFileProperties` 改为读 `KnowledgeFileRuntimeConfigService`：

1. 获取当前运行时配置。
2. 校验文件后缀是否在 `allowedSuffixes` 中。
3. 校验 `file.getSize() <= maxSizeBytes`。
4. 其余上传流程保持不变。

### 7.5 `KnowledgeFileProperties` 角色调整

修改后的角色：

- `allowedSuffixes`、`maxSizeBytes`：仅作为默认值和数据库空配置回退值。
- `internalBaseUrl`、`serviceToken`：继续由 YAML 管理。
- 如后续希望完全解耦，可把“动态规则默认值”和“内部服务参数”拆成两个 Properties 类。

### 8.2 YAML 配置项

继续保留：

```yaml
tolink:
  knowledge-file:
    allowed-suffixes:
      - md
      - markdown
      - pdf
      - docx
      - txt
    max-size-bytes: ${KNOWLEDGE_FILE_MAX_SIZE_BYTES:20971520}
    internal-base-url: ${KNOWLEDGE_FILE_INTERNAL_BASE_URL:http://tolink-service:8080}
    service-token: ${KNOWLEDGE_FILE_SERVICE_TOKEN:}
```

规则说明：

- `allowed-suffixes`、`max-size-bytes` 不再是管理员日常修改入口。
- 它们作为初始化默认值和数据库兜底值。
- `internal-base-url`、`service-token` 仍由部署环境维护。
