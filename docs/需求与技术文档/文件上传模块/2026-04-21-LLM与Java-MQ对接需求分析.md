# LLM 与 Java MQ 对接需求分析

## 1. 需求概述

### 1.1 背景

当前 ToLink Java 管理端已经具备知识文件上传、数据集归属、解析任务投递的基础能力，但 Java 与 Python/RAG 之间的 MQ 对接只完成了 `parse_task` 发送端，尚未形成完整的异步解析闭环。

现有问题主要有：

- Java 端只有解析任务发送，没有解析结果消费闭环。
- `parse_result` 回写规则目前只存在于对接说明文档中，未沉淀为正式需求约束。
- MQ 一期范围与二期范围尚未在需求层明确，容易造成实现扩散。
- 异步消费失败后的补偿要求尚未固化，后续容易出现“能消费但不可恢复”的半成品链路。

### 1.2 目标

- 明确 Java 与 Python/RAG 在 MQ 链路中的一期对接范围，只建设文件解析闭环。
- 明确 `parse_task` 与 `parse_result` 的业务职责、消息字段、状态回写和异常处理要求。
- 明确 MQ 消费失败后的重试与补偿要求，为后续技术实现提供验收依据。

### 1.3 成功标准

- Java 端能稳定投递 `parse_task` 消息。
- Python/RAG 能按约定消费 `parse_task` 并回传 `parse_result`。
- Java 端能按 `task_id` 正确消费 `parse_result` 并回写 `document_original_file`。
- 解析结果消费失败时，系统具备明确的重试和补偿机制要求，而不是仅靠人工排查日志。

## 2. 用户与场景

### 2.1 用户角色

| 角色 | 说明 | 权限边界 |
| --- | --- | --- |
| 普通用户 | 上传文件、发起解析、查看解析状态 | 只能操作自己的数据集、对话和文件 |
| Java 服务 | 负责投递解析任务、消费解析结果、维护业务状态 | 负责业务表、权限校验和状态回写 |
| Python/RAG 服务 | 负责消费解析任务、执行解析、回传解析结果 | 不直接写 Java 业务库，只通过 MQ 回传结果 |
| 运维/管理员 | 处理消费异常、补偿失败任务 | 只处理系统异常，不直接参与用户业务操作 |

### 2.2 使用场景

| 场景 | 触发条件 | 用户目标 |
| --- | --- | --- |
| 文件上传后立即解析 | 用户上传文件并选择立即解析 | 尽快得到解析结果 |
| 文件上传后稍后解析 | 用户先上传文件，后续再手动触发解析 | 按批次控制解析时机 |
| 解析结果回写 | Python/RAG 完成解析后回传结果消息 | Java 端正确更新文件解析状态 |
| 解析消费失败补偿 | `parse_result` 消费失败或状态未回写成功 | 能重试或补偿，不丢任务 |

## 3. 范围定义

### 3.1 必须支持

- 一期只支持文件解析闭环消息：
  - `tolink.rag.parse_task`
  - `tolink.rag.parse_result`
- 一期联调与验收基线使用 Kafka。
- Java 端在文件满足解析条件后投递 `parse_task` 消息。
- Python/RAG 端消费 `parse_task` 后执行解析，并通过 `parse_result` 回传结果。
- Java 端消费 `parse_result` 后按规则回写 `document_original_file`。
- `parse_result` 消费失败时，必须有重试与补偿要求。

### 3.2 可选增强

- 增加 `parse_result` 的消费监控面板或管理端补偿入口。
- 增加结果消息回写统计与异常报表。
- 增加更细粒度的解析阶段状态，如 `processing`、`retrying`。

### 3.3 暂不支持

- 一期不建设 `cache_sync` 消息链路。
- 一期不建设 `usage_report` 消息链路。
- 一期不要求 RabbitMQ 作为联调与验收基线。
- 一期不允许 Python/RAG 直接写 Java 业务数据库。

## 5. 业务流程

### 5.1 主流程

1. 用户在数据集中上传原始文件，并选择立即解析或稍后触发解析。
2. Java 端完成原文件持久化，并在满足条件时投递 `tolink.rag.parse_task`。
3. Python/RAG 消费 `parse_task`，通过 Java 内部文件下载地址获取原文件并执行解析。
4. Python/RAG 解析完成后投递 `tolink.rag.parse_result`。
5. Java 端消费 `parse_result`，按 `task_id` 和 `document_id` 校验后回写 `document_original_file`。
6. 用户在文件列表或详情中查看解析结果状态。

### 5.2 异常流程

| 异常场景 | 期望行为 | 用户提示 |
| --- | --- | --- |
| 原文件上传失败 | 不投递 `parse_task`，保留失败记录 | 文件上传失败，请稍后重试 |
| `parse_task` 投递失败 | 保留原文件记录，标记通知失败 | 文件已上传，解析任务投递失败 |
| Python/RAG 消费 `parse_task` 失败 | 由 Python/RAG 侧按消费策略处理，不直接改 Java 业务状态 | 无直接用户提示 |
| `parse_result` 消费失败 | 必须支持 MQ 重试，并提供补偿能力 | 无直接用户提示，但系统需可恢复 |
| `parse_result` 中 `task_id` 与数据库不匹配 | 拒绝回写并记录错误日志 | 无直接用户提示 |
| 已成功任务收到重复失败结果 | 忽略失败消息，不允许覆盖成功状态 | 无直接用户提示 |

### 5.3 状态流转

| 当前状态 | 触发动作 | 目标状态 | 说明 |
| --- | --- | --- | --- |
| `not_started` | 创建解析任务并成功投递 `parse_task` | `pending` | 表示已进入解析队列 |
| `pending` | Python/RAG 完成解析并回传成功结果 | `success` | 写入解析结果地址与完成时间 |
| `pending` | Python/RAG 回传失败结果 | `failed` | 写入解析失败原因 |
| `failed` | 后续收到同 `task_id` 的成功结果 | `success` | 允许成功覆盖失败 |
| `success` | 收到重复成功或失败结果 | `success` | 保持成功，不允许被覆盖 |

## 6. 功能需求

### 6.1 功能清单

| 功能点 | 描述 | 优先级 | 验收方式 |
| --- | --- | --- | --- |
| 解析任务消息投递 | Java 将符合条件的文件投递到 `tolink.rag.parse_task` | P0 | 联调验证消息体和 broker 中消息 |
| 解析结果消息消费 | Java 消费 `tolink.rag.parse_result` 并回写文件状态 | P0 | 联调验证数据库状态变化 |
| 幂等校验 | Java 以 `task_id` 为主幂等键，结合 `document_id` 做辅助校验 | P0 | 重复消息测试 |
| 成功回写 | 成功结果更新解析成功状态和结果地址 | P0 | 数据库断言 |
| 失败回写 | 失败结果更新解析失败状态和失败原因 | P0 | 数据库断言 |
| 消费失败补偿 | `parse_result` 消费失败必须支持重试和补偿恢复 | P0 | 异常场景演示或测试 |
| 二期消息规划 | `cache_sync`、`usage_report` 只在需求中保留规划 | P1 | 文档评审 |

### 6.2 业务规则

- 一期只建设 `parse_task` 与 `parse_result` 两类 MQ 消息。
- 一期 Kafka 为唯一联调与验收基线，RabbitMQ 仅保留组件兼容能力。
- Java 与 Python/RAG 必须连接同一 MQ Broker，并遵守统一 Envelope 结构。
- `parse_task` 必须包含：
  - `task_id`
  - `document_id`
  - `file_url`
  - `file_type`
- `parse_result` 必须包含：
  - `task_id`
  - `document_id`
  - `success`
  - `status`
  - 成功时的解析结果地址字段
  - 失败时的失败原因字段
- Java 回写解析结果时：
  - 以 `task_id` 为主幂等键
  - `document_id` 不匹配时拒绝回写
  - 已成功任务不能被失败结果覆盖
  - 已失败任务可以被后续成功结果覆盖
- Python/RAG 不得直接写 Java 业务数据库。

### 6.3 权限规则

| 操作 | 普通用户 | 管理员 | 未登录 |
| --- | --- | --- | --- |
| 上传并触发解析 | 允许，仅限本人数据集 | 允许 | 不允许 |
| 查看解析状态 | 允许，仅限本人文件 | 允许 | 不允许 |
| MQ 消息生产/消费 | 不涉及 | 不涉及 | 不涉及 |
| 异常任务补偿 | 不允许 | 允许 | 不允许 |

## 7. 数据需求

### 7.1 需要保存的数据

| 数据项 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `parse_task_id` | string | 是 | 解析任务唯一标识，Java 与 Python/RAG 共享 |
| `document_id` | long | 是 | Java 原文件记录 ID |
| `parse_status` | string | 是 | 解析状态，如 `not_started/pending/success/failed` |
| `is_parse_success` | boolean | 是 | 是否解析成功 |
| `parsed_bucket_name` | string | 否 | 成功时的解析结果 bucket |
| `parsed_object_key` | string | 否 | 成功时的解析结果 object key |
| `parsed_file_url` | string | 否 | 成功时的解析结果访问地址 |
| `parse_failure_reason` | string | 否 | 解析失败原因 |
| `parsed_at` | datetime | 否 | 解析成功回写时间 |

### 7.2 数据约束

- `parse_task_id` 在 `document_original_file` 中必须唯一。
- `parse_status` 必须受限于系统允许的状态集合。
- 成功结果写入后，结果地址字段必须完整。
- 失败结果写入时，不得清空已存在的成功结果地址字段。

### 7.3 数据留存与隐私

- 敏感数据：内部文件下载地址、解析结果对象地址。
- 是否需要脱敏：日志中不应输出敏感认证信息。
- 是否需要审计：需要记录消息消费成功、失败、忽略、拒绝回写等关键事件。
- 是否需要删除或归档策略：遵循文件主记录的生命周期，不单独建立解析任务表。

## 8. API 需求

本需求的一期核心是 MQ 对接，不新增对外用户 API，但需明确内部能力边界。

| 接口能力 | 调用方 | 输入 | 输出 | 权限 |
| --- | --- | --- | --- | --- |
| 内部原文件下载 | Python/RAG | `document_id`、`task_id` | 原文件流 | 内部服务调用 |
| 文件状态查询 | 前端/用户 | 文件 ID 或数据集维度筛选 | 当前解析状态、失败原因、结果地址 | 登录用户 |
| 异常任务补偿能力 | 管理端/运维 | `task_id` 或文件标识 | 补偿执行结果 | 管理员 |

## 9. 非功能需求

| 类型 | 要求 |
| --- | --- |
| 性能 | MQ 通信必须支持异步解耦，避免文件上传链路同步等待解析完成 |
| 安全 | Python/RAG 不直接访问 Java 业务库；内部文件下载必须受 `task_id` 或内部凭证保护 |
| 兼容性 | 一期以 Kafka 为准，但消息模型需兼容现有 MQ 抽象 |
| 可观测性 | 必须记录消息投递、消费成功、消费失败、幂等忽略、拒绝回写等日志 |
| 可维护性 | 二期消息类型需可在当前 MQ 抽象上继续扩展 |

## 10. 验收标准

| 编号 | 验收项 | 验收方式 |
| --- | --- | --- |
| AC-1 | Java 能向 `tolink.rag.parse_task` 投递符合约定结构的消息 | Kafka 联调验证 |
| AC-2 | Python/RAG 回传成功结果后，Java 能正确回写文件解析成功状态 | 数据库与接口联调验证 |
| AC-3 | Python/RAG 回传失败结果后，Java 能正确回写解析失败状态与失败原因 | 数据库与接口联调验证 |
| AC-4 | 重复成功消息不会导致错误覆盖 | 幂等测试 |
| AC-5 | 成功状态不会被后续失败消息覆盖 | 乱序测试 |
| AC-6 | `task_id` 不匹配的结果消息会被拒绝回写并记录异常日志 | 异常测试 |
| AC-7 | `parse_result` 消费失败时，系统具备重试与补偿能力 | 异常流程验证 |
| AC-8 | `cache_sync` 与 `usage_report` 不进入一期交付范围 | 需求评审 |

## 11. 假设与依赖

### 11.1 假设

- Java 与 Python/RAG 使用同一 MQ Broker。
- Python/RAG 侧能够按约定访问 Java 内部文件下载接口。
- Python/RAG 侧能够保证 `parse_result` 中 `task_id` 与 `document_id` 的正确性。

### 11.2 外部依赖

- Kafka Broker 可用且配置正确。
- Python/RAG 服务具备对应 topic 的生产和消费能力。
- Java 侧已有统一 MQ 抽象组件可继续复用。
