## 2. 技术目标与范围 (Overview)

### 2.1 技术目标 (Technical Goals)

* **核心目标：**
  - 在现有 `document_original_file` 之外新增一张独立表，专门保存 Python 端返回的解析结果信息。
  - Java 端继续只负责消费 MQ 并存储结果，不新增任何“文件解析”或“解析产物生成”逻辑。
  - 保持现有上传与 `parse_result` 消费主链路稳定，并将原表收缩为“原文件上传信息 + 任务投递信息”最小模型。

* **成功标准：**
  - 成功或失败的 `parse_result` 消息都能落到独立结果表。
  - 每个 `document_original_file` 最多只对应一条当前解析结果记录。
  - `document_original_file` 不再存储解析结果明细字段，仅保留必要任务追踪字段。
  - 结果表支持按原文件、数据集、状态独立查询。

### 2.2 实现范围 (In Scope / Out of Scope)

**必须实现：**

- 新增解析结果独立存储表。
- 新增对应 Entity、Mapper。
- 在现有 `KnowledgeParseResultServiceImpl` 中增加结果表 upsert 逻辑，并停止回写原表解析结果字段。
- 删除原表中的冗余解析结果字段。
- 补充结果表的一对一约束与必要索引。
- 补充成功、失败、重复消息、失败转成功的测试设计与实现。

**暂不实现：**

- 不让 Java 参与文件解析内容生成。
- 不让 Python 端直连数据库写结果表。
- 不保留解析历史版本。
- 不支持一个原文件对应多条解析结果记录。
- 不新增管理端写接口，只设计查询能力。

## 4. 总体方案设计 (Architecture & Solution)

### 4.1 总体设计思路

采用“保留现有 MQ 消费链路 + 在消费服务中增加结果表持久化”的最小改造策略：

- `parse_result` 的 Kafka listener、消息模型、业务 receiver 全部保持不变。
- `KnowledgeParseResultServiceImpl` 继续作为唯一业务入口。
- 该服务在处理消息时执行一个核心动作：
  - 按 `document_original_file_id` 对解析结果表执行插入或更新
- `KnowledgeFileServiceImpl` 在发送 `parse_task` 成功后，提前写入 `pending` 解析记录。
- `document_original_file` 不再维护解析结果明细，只保留任务投递字段。

这样可以保证：

- 原表职责清晰，只承载原文件与任务派发信息
- 解析结果查询统一从新表读取
- 改造范围集中，不引入第二套消费链路

### 4.2 目标调用链路

```text
Python/RAG
  -> Kafka(tolink.rag.parse_result)
  -> KnowledgeParseResultKafkaReceiver
  -> KnowledgeParseResultConsumer
  -> KnowledgeParseResultServiceImpl
     -> KnowledgeParsedFileMapper.selectOne/insert/updateById(...)
  -> MySQL
```

### 4.3 核心模块职责划分

| 模块/类 | 职责 | 输入/输出边界 |
| :--- | :--- | :--- |
| `KnowledgeParseResultMQ` | 定义结果消息 payload | 输入 MQ JSON，输出 payload |
| `KnowledgeParseResultKafkaReceiver` | Kafka 消息接收 | 输入字符串消息 |
| `KnowledgeParseResultConsumer` | 业务 receiver | 输入 payload |
| `KnowledgeParseResultServiceImpl` | 校验、幂等、upsert 结果表 | 输入 payload，输出无 |
| `KnowledgeFileServiceImpl` | 创建解析任务并预建 `pending` 结果记录；列表/详情联动读取解析表 | 输入上传/查询请求，输出 DTO |
| `KnowledgeParsedFile` | 解析结果独立实体 | 输入数据库行，输出实体对象 |
| `KnowledgeParsedFileMapper` | 解析结果表持久化 | 输入查询条件/实体，输出数据库结果 |

### 4.4 关键设计决策

1. 新表命名采用 `document_parsed_file`

- 与现有 `document_original_file` 命名风格一致
- 明确表达“原文件解析后对应的结果文件信息”

2. 删除原表解析结果明细字段，避免双份事实来源

- 解析状态和结果信息只在新表维护
- 避免原表与结果表长期漂移

3. 结果表按 `document_original_file_id` 做唯一约束

- 与需求“一个原文件只对应一个当前解析结果”严格对齐
- 比单独依赖 `parse_task_id` 更贴近业务主键

4. `parse_task_id` 在结果表中也保留

- 便于按 MQ 任务追踪
- 便于与原文件任务记录交叉校验

本轮不强制新增新 service，优先在现有 `KnowledgeParseResultServiceImpl` 中完成结果表 upsert。

可选拆分方式：

```java
public interface KnowledgeParsedFilePersistenceService {

    void saveOrUpdateFromParseResult(KnowledgeOriginalFile originalFile,
                                     KnowledgeParseResultMQ.MsgPayload payload,
                                     boolean parseSuccess);
}
```

若本轮追求最小改动，也可以暂时作为 `KnowledgeParseResultServiceImpl` 的私有方法保留。

### 7.3 核心方法职责

| 方法 | 职责 | 输入 | 输出 |
| :--- | :--- | :--- | :--- |
| `handleParseResult` | 校验消息、幂等处理、协调结果表 upsert | `MsgPayload` | `void` |
| `handleSuccess` | 更新结果表成功字段 | 原文件记录 + payload | `void` |
| `handleFailure` | 更新结果表失败字段 | 原文件记录 + payload | `void` |
| `saveOrUpdateParsedFile` | 按 `document_original_file_id` upsert 结果表 | 原文件记录 + payload | `void` |

### 7.4 关键业务流程

1. 按 `parse_task_id` 查询 `document_original_file`
2. 校验 `document_id` 是否匹配
3. 查询 `document_parsed_file` 是否已存在
4. 若结果表已是 `success`，对重复成功/失败消息执行幂等忽略
5. 不存在则插入，存在则更新
6. 成功时写入 md 文件信息；失败时清空成功字段并写失败原因

### 7.5 建议的 upsert 规则

成功消息：

- 更新 `parse_status=success`
- 更新 `is_parse_success=true`
- 更新 `parsed_filename`
- 更新 `parsed_bucket_name`
- 更新 `parsed_object_key`
- 更新 `parsed_file_url`
- 更新 `parsed_storage_path`
- 更新 `parse_result`
- 清空 `failure_reason`
- 更新 `parsed_at`、`last_result_at`

失败消息：

- 更新 `parse_status=failed`
- 更新 `is_parse_success=false`
- 清空 `parsed_filename`
- 清空 `parsed_bucket_name`
- 清空 `parsed_object_key`
- 清空 `parsed_file_url`
- 清空 `parsed_storage_path`
- 可选清空 `parse_result` 或写入失败结果描述，推荐保留 Python 返回的失败结果描述
- 更新 `failure_reason`
- 清空 `parsed_at`
- 更新 `last_result_at`

说明：以上字段全部作用于 `document_parsed_file`，不再回写 `document_original_file`。

## 8. 组件与集成设计 (Integration Design)

| 组件 | 用途 | 配置项 | 失败处理 |
| :--- | :--- | :--- | :--- |
| Kafka | 接收 `parse_result` 消息 | 复用现有 `tolink.mq.parse-result.group-id` | 消费异常按现有 listener 异常传播与日志处理 |
| MySQL | 存储原文件与结果表 | 无新增中间件 | 事务内结果表写失败则整体回滚 |
| OSS/MinIO | 存储 Python 生成的 md 文件 | 无 Java 新配置 | Java 只保存地址，不直接访问或上传 |

### 8.1 事务边界

- `KnowledgeParseResultServiceImpl.handleParseResult` 保持 `@Transactional`
- 原文件查询与结果表 upsert 放在同一事务中
- 结果表写失败，则整个消费处理失败，由 MQ 重试或人工排查

### 8.2 幂等与一致性

- 原表按 `parse_task_id` 定位原文件记录
- 结果表按 `document_original_file_id` 唯一定位记录
- 已成功记录收到后续失败消息时继续忽略，保持成功结果不被脏覆盖
- 已失败记录收到后续成功消息时允许覆盖为成功

### 8.3 日志要求

建议补充以下日志：

- 结果表插入成功：`task_id`、`document_id`、`dataset_id`
- 结果表更新成功：`task_id`、`document_id`、`parse_status`
- 幂等忽略：`task_id`、`document_id`、`current_status`
- 历史迁移阶段发现原表旧字段与新表不一致时：打印 `document_original_file_id`、`parse_task_id`

## 10. 风险与待确认项 (Risks)

### 10.1 主要风险

- `parse_result` 协议当前未明确 `parse_result` 字段的长度和语义，过长内容可能不适合直接入库 `text`
- `parsed_filename` 是否总能由 Python 返回，若不能则需从 `parsed_object_key` 推导
- 原文件删除后的结果表处理策略仍需产品确认

### 10.2 低风险待确认项

| 编号 | 问题 | 影响 |
| :--- | :--- | :--- |
| Q-1 | 原文件删除时，结果表是物理删除还是逻辑不可见 | 影响数据清理策略 |
| Q-2 | `parse_result` 最终保存完整文本、摘要文本还是固定状态描述 | 影响字段长度与接口展示 |
