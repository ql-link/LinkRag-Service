## 3. 当前数据流

### 3.1 创建对话

1. 前端提交 `datasetId`
2. `ChatServiceImpl` 校验当前用户是否拥有该数据集
3. 写入 `chat_conversation.dataset_id`
4. 后续消息检索通过 `conversationId -> datasetId` 间接限定知识范围

### 3.2 上传文件

1. 前端在数据集上下文下上传文件
2. `KnowledgeFileServiceImpl` 校验：
   - 数据集归属
   - 文件后缀
   - 文件大小
   - 文件名合法性
   - 同数据集下是否重名
3. 先插入上传中记录
4. 调用 `IOssService` 上传到私有存储
5. 更新 `document_original_file` 为上传成功
6. 若 `parseImmediately=true`，通过 MQ 投递解析任务

### 3.3 删除文件

1. 先根据 `fileId + userId` 校验归属
2. 若存在 `objectKey`，先删除 OSS 原始对象
3. 清理 `PrivateFileResolver` 本地缓存
4. 物理删除 `document_original_file`

说明：

- 当前删除顺序是“先 OSS，后数据库”
- 若 OSS 删除失败，数据库记录保持不动
- 若 OSS 删除成功但数据库删除失败，需要通过日志和后续补偿修复不一致

### 3.4 删除数据集

1. 校验数据集归属
2. 查询该数据集全部原始文件并删除 OSS 对象
3. 删除该数据集下所有消息
4. 删除该数据集下所有对话
5. 删除该数据集下所有原始文件记录
6. 删除数据集本身

说明：

- 当前实现优先保证数据库主记录不要先于 OSS 清理被删除
- 任一 OSS 文件删除失败，整次数据集删除会中断
- 若 OSS 清理完成但数据库事务失败，需依赖后续补偿继续清理数据库记录

## 4. OSS 规则

### 4.1 对象存储位置

- OSS 桶名由 [IOssService.java](/Users/fang/Developer/Projects/toLink/toLink-Service/link-components/toLink-components-oss/src/main/java/com/qingluo/link/components/oss/service/IOssService.java) 按存储类型解析
- 原始知识文件统一使用私有存储位置 `OssSavePlaceEnum.PRIVATE`

### 4.2 对象 Key 规则

当前 [KnowledgeFileServiceImpl.java](/Users/fang/Developer/Projects/toLink/toLink-Service/link-service/src/main/java/com/qingluo/link/service/impl/KnowledgeFileServiceImpl.java) 中实际生成规则为：

```text
{userId}/{datasetId}/{yyyy}/{MM}/{dd}/{originalFilename}
```

结合业务约定，完整语义路径为：

```text
rag-raw/{userId}/{datasetId}/{yyyy}/{MM}/{dd}/{originalFilename}
```

说明：

- `rag-raw` 是业务前缀语义
- 代码里当前写入 `object_key` 的是后半段相对路径
- 实际桶名与前缀组合方式由具体 OSS 实现和部署配置决定

## 5. 当前约束

- 不支持同一数据集下同名文件自动覆盖
- 不支持同一对话绑定多个数据集
- 不保留软删除记录
- 不建设 `document_parse_task` 表
- 解析结果状态只回写 `document_original_file`
