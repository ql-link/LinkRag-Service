# 解析文件独立存储表需求分析

## 1. 需求概述

### 1.1 背景

当前知识文件上传与解析链路中，原文件上传记录、解析任务状态、解析结果文件地址都集中保存在 `document_original_file` 中。该方案可以满足一期快速闭环，但随着数据集中心化模型落地，Python 端返回的解析结果信息已经具备了独立业务语义：

- 原始文件是上传对象，关注上传过程与原文件存储信息
- 解析结果信息是 Python 端回传给 Java 的结果对象，关注结果状态、结果文件信息及失败信息

继续将解析结果字段作为原文件表快照，会带来以下问题：

- 原文件记录与解析文件记录语义混杂，不利于后续扩展
- 无法以“解析文件”为独立对象进行查询、约束和后续管理
- 失败结果只能以零散字段方式附着在原文件表上，不利于统一处理

因此需要新增一张“解析文件信息表”，用于独立保存 Python 端返回的解析结果信息，以及解析失败记录。Java 端只负责接收并存储这些结果信息，不负责文件解析过程，也不负责解析产物内容生成。

同时，为了让表职责更单一，`document_original_file` 应尽量只保留“原文件上传信息”和“解析任务投递信息”，不再长期保留解析结果明细字段。

### 1.2 目标

- 将 Python 端返回的解析结果信息从 `document_original_file` 中拆分为独立业务对象
- 支持保存解析成功和解析失败两类结果
- 明确“一个原文件只对应一个当前解析结果记录”的业务规则
- 明确该表至少保存原文件主键、原文件名、解析后存储位置、解析结果

### 1.3 成功标准

- 系统能够为每个原文件保存一条独立的解析结果记录
- 解析成功时可查询到 md 文件存储信息
- 解析失败时可查询到失败状态和失败原因
- 独立查询解析结果时，无需依赖扫描 `document_original_file` 中的 `parsed_*` 快照字段

## 2. 用户与场景

### 2.1 用户角色

| 角色 | 说明 | 权限边界 |
| --- | --- | --- |
| 普通用户 | 上传知识文件、查看文件解析结果 | 只能查看自己数据集下原文件对应的解析结果 |
| 管理员 | 排查问题、审计解析结果、辅助补偿 | 可查看全量解析结果，是否允许跨用户手工修改由后续技术设计决定 |
| Python/RAG 服务 | 解析任务执行方，负责解析并返回结果信息 | 不直接写业务库，通过 MQ 回传解析结果 |
| Java 服务 | 解析结果消费方，负责接收并落库 | 只存储 Python 返回的信息，不负责解析和产物生成 |

### 2.2 使用场景

| 场景 | 触发条件 | 用户目标 |
| --- | --- | --- |
| 用户查看某个原文件的解析结果 | 文件已提交解析 | 查看 Python 返回的解析状态，以及成功后的 md 文件存储信息 |
| 用户排查解析失败 | 文件解析失败 | 查看失败原因并决定是否重试 |
| 后台按条件查询解析结果 | 管理员或系统排障 | 按数据集、原文件、任务状态查询解析记录 |

## 3. 范围定义

### 3.1 必须支持

- 新增独立的解析文件信息表
- 每个原文件最多对应一条当前解析结果记录
- 解析成功时保存 Python 返回的 md 文件信息
- 解析失败时保存失败状态和失败原因
- 支持独立查询解析结果记录
- 支持通过约束保证原文件与解析结果的一对一关系
- 表内必须包含原文件主键、原文件名、解析后存储位置、解析结果等核心信息
- 原文件表应删除可被解析结果表替代的冗余解析字段，保持模型简洁

### 3.2 可选增强

- 增加按数据集、用户、解析状态的筛选查询能力
- 增加管理端解析结果排查页面或接口
- 为后续补偿任务提供查询入口

### 3.3 暂不支持

- 一个原文件对应多份解析结果记录
- 保存历史版本
- 保存多个结果类型，如 `json`、`chunk`、`image manifest`
- Python 端直连数据库写入结果表

## 5. 业务流程

### 5.1 主流程

1. 用户上传原文件，系统先在 `document_original_file` 中创建原文件记录。
2. Java 服务向 Python/RAG 侧投递解析任务。
3. Python/RAG 侧完成解析后，将解析结果通过 MQ 回传给 Java。
4. Java 服务消费解析结果后，在解析文件信息表中新增或更新该原文件对应的解析结果记录。
5. 原文件表仅保留原文件上传信息与解析任务投递信息，不再保存解析结果明细。
6. 用户后续可单独查询该解析结果记录。

### 5.2 异常流程

| 异常场景 | 期望行为 | 用户提示 |
| --- | --- | --- |
| Python 解析失败 | 仍然落一条解析结果记录，状态标记为失败，保存 Python 返回的失败原因 | 文件解析失败，请稍后重试 |
| MQ 重复投递相同结果 | 不得生成多条解析结果记录，应按原文件唯一键执行幂等更新 | 不直接提示用户 |
| 同一原文件先失败后成功 | 不新增历史版本，更新同一条解析结果记录为最新状态 | 页面展示最终最新状态 |
| 查询不存在的解析结果 | 返回“无解析结果”或“尚未产生解析结果” | 暂无解析结果 |

### 5.3 状态流转

| 当前状态 | 触发动作 | 目标状态 | 说明 |
| --- | --- | --- | --- |
| 不存在 | Java 创建解析任务 | 待解析 | Java 在任务发出成功后提前创建 `pending` 记录 |
| 待解析 | Python 返回成功结果 | 解析成功 | 保存 Python 返回的 md 文件信息 |
| 待解析 | Python 返回失败结果 | 解析失败 | 保存 Python 返回的失败原因 |
| 解析失败 | 重新解析并成功 | 解析成功 | 复用同一条记录，不保留历史版本 |
| 解析成功 | 重复成功消息 | 解析成功 | 幂等忽略或覆盖相同内容，不能新增第二条记录 |

## 6. 功能需求

### 6.1 功能清单

| 功能点 | 描述 | 优先级 | 验收方式 |
| --- | --- | --- | --- |
| 独立解析结果建模 | 新增解析文件信息表，承载 Python 返回的解析结果对象语义 | P0 | 查看表结构与文档 |
| 成功结果保存 | 成功时保存 Python 返回的 md 文件 bucket、objectKey、fileUrl 等信息 | P0 | 结果回写验证 |
| 失败结果保存 | 失败时保存 Python 返回的失败状态、失败原因 | P0 | 失败场景验证 |
| 一对一约束 | 一个原文件最多一条当前解析结果记录 | P0 | 唯一约束验证 |
| 独立查询 | 可单独查询解析结果记录，不依赖原文件表中的快照字段 | P0 | 查询接口或查询能力验证 |

### 6.2 业务规则

- 一个 `document_original_file` 只允许对应一条当前解析结果记录。
- Java 端只负责存储 Python 返回的解析结果信息，不负责解析文件内容、不负责解析产物生成。
- Python 端解析成功后，只会返回一份 md 文件对应的结果信息。
- 解析结果表中既要覆盖成功结果，也要覆盖失败结果。
- 不保留历史版本；同一原文件再次解析时，应更新原记录，而不是插入新记录。
- 解析结果表中必须能够直接拿到原文件主键、原文件名、解析后存储位置、解析结果，避免查询时必须二次拼装最基础信息。
- 原文件表只保留上传信息和解析任务投递信息；解析状态、成功标记、结果地址、失败原因等由解析结果表独立承担。
- 解析成功时，Python 返回的 md 文件存储信息为必填。
- 解析失败时，Python 返回的失败原因为必填，md 文件存储信息允许为空。
- 原文件删除后的解析结果处理策略需要与原文件生命周期保持一致，默认应同步删除或不可见。

### 6.3 权限规则

| 操作 | 普通用户 | 管理员 | 未登录 |
| --- | --- | --- | --- |
| 查询本人数据集下原文件的解析结果 | 允许 | 允许 | 不允许 |
| 查询其他用户数据集下解析结果 | 不允许 | 允许 | 不允许 |
| 新增/更新解析结果 | 不允许 | 不直接人工操作 | 不允许 |

## 7. 数据需求

### 7.1 需要保存的数据

| 数据项 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 是 | 解析结果记录主键 |
| document_original_file_id | bigint | 是 | 关联原文件记录 ID |
| original_filename | varchar(255) | 是 | 原文件名快照，便于独立查询展示 |
| dataset_id | bigint | 是 | 冗余保存所属数据集 ID，便于独立查询 |
| user_id | bigint | 是 | 冗余保存所属用户 ID，便于权限过滤 |
| parse_task_id | varchar(36) | 是 | 解析任务业务唯一标识 |
| parse_status | varchar(16) | 是 | 解析状态，如 `pending/success/failed` |
| is_parse_success | tinyint(1) | 是 | 是否解析成功 |
| parsed_filename | varchar(255) | 否 | Python 返回的 md 文件名，失败时可为空 |
| parsed_bucket_name | varchar(64) | 否 | Python 返回的 md 文件桶名，失败时可为空 |
| parsed_object_key | varchar(512) | 否 | Python 返回的 md 文件对象 Key，失败时可为空 |
| parsed_file_url | varchar(1024) | 否 | Python 返回的 md 文件访问地址或内部下载地址，失败时可为空 |
| parsed_storage_path | varchar(1024) | 否 | 解析后文件存储位置，可由 bucket + objectKey 组合或由协议直接返回 |
| parse_result | text | 否 | Python 返回的解析结果摘要或结果描述，成功失败均可保存 |
| failure_reason | varchar(512) | 否 | Python 返回的解析失败原因，成功时为空 |
| parsed_at | datetime | 否 | 解析成功时间 |
| last_result_at | datetime | 是 | 最近一次收到解析结果时间 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 更新时间 |

### 7.2 数据约束

- `document_original_file_id` 必须唯一，保证一对一关系。
- `parse_task_id` 应具备唯一性或至少具备唯一索引约束，避免任务结果混淆。
- `dataset_id`、`user_id` 应与原文件记录中的归属保持一致。
- `original_filename` 应保存为原文件名快照，即使原表后续变化，独立查询仍能直接展示。
- `parsed_storage_path` 作为解析后存储位置的统一展示字段，可与 `parsed_bucket_name`、`parsed_object_key` 并存。
- `parse_result` 用于保存 Python 返回的解析结果信息；若结果内容较长，技术方案阶段需明确长度和展示策略。
- 当 `is_parse_success=true` 时，`parsed_bucket_name`、`parsed_object_key`、`parsed_file_url` 必填。
- 当 `is_parse_success=false` 时，`failure_reason` 必填。
- 同一原文件重复收到结果消息时，应执行更新而非新增。

建议从 `document_original_file` 中删除以下冗余解析字段：

- `parse_status`
- `is_parse_success`
- `parsed_bucket_name`
- `parsed_object_key`
- `parsed_file_url`
- `parsed_at`
- `parse_failure_reason`

建议在 `document_original_file` 中保留以下最小解析相关字段：

- `parse_task_id`
- `parse_notice_status`
- `parse_notice_retry_count`
- `last_parse_notice_at`

### 7.3 数据留存与隐私

- 敏感数据：无高敏内容，但 `parsed_file_url` 若为内部访问地址，需避免前端直接暴露内部网络信息。
- 是否需要脱敏：前端展示时按接口场景决定，管理端可看全量，普通用户按权限返回。
- 是否需要审计：建议记录最近一次结果更新时间和任务标识，便于排障。
- 是否需要删除或归档策略：默认与原文件生命周期一致，原文件删除后解析结果应同步删除或同步不可见。

## 9. 非功能需求

| 类型 | 要求 |

| 兼容性 | 应兼容现有 MQ 回写链路；Java 不新增解析逻辑，只新增结果存储能力 |

| 编号 | 验收项 | 验收方式 |
| --- | --- | --- |

## 11. 假设与依赖

### 11.1 假设

- 当前阶段 Python 端每次解析只产出一份 md 文件。
- Python 端不直接写数据库，而是继续通过 MQ 回传结果。
- Java 端只负责消费并存储 Python 返回的信息，不负责解析文件内容和解析产物生成。
- 当前实现中，原文件表已不再保留解析结果字段；解析状态与结果信息统一从 `document_parsed_file` 读取。
- 新表是解析结果主记录，原文件表仅承担原文件上传与任务投递职责。

### 11.2 外部依赖

- Python/RAG 服务按既定消息协议返回解析结果
- Java MQ 消费端负责执行解析结果落库
- OSS/MinIO 正常保存解析后的 md 文件
