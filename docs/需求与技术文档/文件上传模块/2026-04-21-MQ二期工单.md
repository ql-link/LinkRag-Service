# MQ 二期工单

## 1. 工单标题

LLM 与 Java MQ 对接二期完善

## 2. 背景

当前一期已完成以下能力：

- `parse_task` 发送
- `parse_result` 消费与状态回写
- 数据库状态更新与基础幂等处理
- 单元测试、回归测试、集成测试

当前仍有一些未纳入一期范围、但后续正式联调和工程化上线需要补齐的能力，需要统一收口为二期工单。

## 3. 二期目标

- 完成真实 Kafka Broker 下的联调验证
- 完成 Python/RAG 真实对接验证
- 补齐 `parse_result` 消费失败后的补偿能力
- 规划并实现 `cache_sync`、`usage_report` 消息链路
- 补充监控、告警、运维支持能力

## 4. 范围

### 4.1 必做项

1. 真实 Kafka 联调
- 在真实 broker 环境下验证 `parse_task` 和 `parse_result`
- 验证消息发送、消费、重试和状态回写

2. Python/RAG 联调
- 验证 Python/RAG 按约定消费 `parse_task`
- 验证 Python/RAG 按约定回传 `parse_result`
- 验证异常消息、重复消息、乱序消息处理

3. `parse_result` 补偿能力
- 明确补偿方案
- 至少实现一种可执行补偿方式：
  - 管理端人工补偿入口
  - 系统补偿任务

4. `cache_sync` 消息链路
- 明确消息体结构
- 明确 Java 生产端和 Python/RAG 消费端职责
- 完成代码落地和测试

5. `usage_report` 消息链路
- 明确最终方向为 Python/RAG 生产，Java 消费落库
- 完成消息模型、消费逻辑和落库逻辑

6. 可观测性补强
- 补充关键日志
- 补充 MQ 消费失败监控
- 补充必要告警项

### 4.2 非本工单范围

- 不调整一期已经通过验证的数据集中心化模型
- 不重构现有 OSS 组件架构
- 不在二期内引入新的 MQ 中间件 vendor

## 5. 验收标准

1. 真实 Kafka 环境下，`parse_task` 与 `parse_result` 联调通过
2. Python/RAG 与 Java 在真实环境中完成闭环验证
3. `parse_result` 消费失败后具备可执行补偿能力
4. `cache_sync` 消息链路可用
5. `usage_report` 消息链路可用且能正确落库
6. 关键失败场景具备日志、监控或告警支持

## 6. 优先级建议

- P0：真实 Kafka 联调、Python/RAG 联调、`parse_result` 补偿能力
- P1：`cache_sync`
- P1：`usage_report`
- P1：监控告警与运维支持

## 7. 备注

- 本工单为二期收口工单，不影响当前一期开发完成状态
- 一期当前可进入联调与验收阶段
